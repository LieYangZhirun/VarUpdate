# VarUpdate 架构设计文档导读

本目录存放 VarUpdate 的设计说明，与 `src/` 实现、`VarUpdate/README.md` 配合使用。

## 各文件职责（按阅读顺序）

| 顺序 | 文件 | 用途 |
|------|------|------|
| 1 | **接口与契约集中定义.md** | 事件名、负载类型、宿主 API、模块对外形状；与实现冲突时以本文档为协商基准，并应回写代码或修正本文档。 |
| 2 | **面向用户功能卡.md** | 作者与用户可见能力：标签、Schema 语法、宏、条件标签等。 |
| 3 | **基础设施功能卡.md** | 各技术模块的职责、数据流与接口摘要（比契约文档更偏「说明文」）。 |
| 4 | **代码规范与工程细节.md** | 构建、TypeScript、全局注入、错误与通知约定。 |
| 5 | **测试规范.md** | 测试分层、目录约定、与模块的对应关系。 |

## 权威层级

1. **运行时行为**：以 `src/**/*.ts` 与 `src/types/index.ts` 为准。  
2. **跨脚本与对外承诺**：以 **接口与契约集中定义.md** 为准；功能卡若与契约有出入，应改功能卡或升级契约并同步代码。  
3. **仓库根目录 README.md**：项目入口与目录树；模块编号与 `架构设计` 不一致时，以契约中的模块章节为准并更新 README。

## 与 Agents 仓库的同步

根目录 **Agents/架构设计/接口与契约集中定义.md** 与本文 **接口与契约集中定义.md** 中，凡涉及**共享事件名、负载类型、宿主 API** 的段落须保持一致；任一侧修改后，另一侧应做相同编辑（两文件开头「维护」段已有说明）。

## 近期已与实现对齐的约定（便于审阅时对照）

- **`MessageCompletePayload.writeMode`**：可选；VarUpdate 不因该字段改变变量写入语义，通知等级为 **debug** 时对非空值输出 **trace**。下标与宿主事件的 **notify** 约定见《接口与契约》「VarUpdate 双通道消息处理」。  
- **`$type` 的 `(force)` 后缀**：仅 `number(force)`、`integer(force)`、`string(force)` 三种为内置语义；不存在 `boolean(force)`。实现上见 `src/shared/schema-force-primitives.ts` 与 `schema-to-zod.ts`。

## 事件名：契约正文与代码

《接口与契约》§1.1 顶部的 `EVENTS` 常量侧重 **跨脚本（Agents ↔ VarUpdate）** 命名空间。VarUpdate 还监听 **酒馆原生事件**（如 `message_received`、`chat_id_changed` 等），完整列表以源码 **`src/modules/event-bus.ts`** 中的 `EVENTS` 为准。
