# 变量更新脚本（VarUpdate）——基础设施功能卡

> 本文档定义 VarUpdate 脚本的底层基础设施模块——支撑面向用户功能卡中所有用户可见功能所需的技术组件、算法和工程化方案。

---

## 一、技术栈与工程化

### 技术选型

| 类别 | 选择 | 说明 |
|------|------|------|
| 语言 | TypeScript | 类型安全，Schema 编译器核心需求 |
| 打包 | esbuild | 快速单文件输出，原生 TS 支持 |
| YAML 解析 | js-yaml | 轻量（~60KB min），API 简洁 |
| TOML 解析 | smol-toml | 最轻量纯 JS TOML 解析器（~15KB min） |
| Schema 校验 | Zod | 运行时类型校验，面向用户功能卡指定 |

| 通知弹窗 | toastr | SillyTavern 全局已引入 |

## 二、模块 1：多格式解析器

> 支撑功能：Schema 定义（二·A-1）、Default 定义（三·B-1）、Initial 初始化（四·C-1）

### 职责

接收一段结构化文本，自动检测格式（YAML / JSON / TOML），解析为 JavaScript 对象。

### 接口

```typescript
/**
 * 将文本解析为 JavaScript 对象
 * @param text 待解析的文本（YAML / JSON / TOML 格式）
 * @returns 解析后的对象
 * @throws FormatParseError 三种格式均无法解析时
 */
function parseStructuredText(text: string): Record<string, any>;
```

### 格式检测策略

按以下顺序尝试解析，首个成功的结果即为最终结果：

1. **JSON**：调用 `JSON.parse(text)`。成功则返回。
2. **TOML**：调用 `smol-toml` 解析。TOML 语法与 YAML 不兼容（TOML 使用 `[section]` 和 `key = value`），可明确区分。
3. **YAML**：调用 `js-yaml` 解析。YAML 是最宽松的格式，作为兜底。

顺序为 JSON → TOML → YAML：合法 JSON 若先走 YAML 无法与「纯 YAML」区分；先 JSON 可走最快路径，TOML 与 YAML 语法互不混淆。

### 错误处理

- 三种格式均解析失败时，抛出 `FormatParseError`，携带三种解析器各自的错误信息，便于用户定位问题。
- 解析成功但结果不是对象类型（如纯字符串、纯数字）时，包裹为 `{ _value: result }` 并附带警告。

## 三、模块 2：Schema 编译器

> 支撑功能：Schema 结构定义（二·A-1 ~ A-6）

### 职责

将声明式 Schema 定义（经模块 1 解析后的 JSON 对象）编译为运行时 Zod 校验对象。核心编译逻辑实现于 `schema-to-zod.ts`（类型映射、`$defs` 解析、约束处理），`index.ts` 作为封装层，负责编译结果的缓存管理、通知系统集成、以及 `bindSafeParseWithContext`（为 refer() 注入运行时上下文的校验器绑定函数）。

### 接口

```typescript
/**
 * 编译结果
 */
interface CompiledSchema {
  /** 编译后的 Zod 校验对象（接受变量状态对象，返回校验结果） */
  validator: z.ZodType<any>;
  /** Schema 原始声明（纯 JSON，用于持久化到 chat 层） */
  raw: Record<string, any>;
  /** $defs 中定义的结构体名称列表（供调试/日志使用） */
  defNames: string[];
}

/**
 * 将 Schema 声明编译为 Zod 校验对象
 * @param schemaData 经格式解析后的 Schema JSON 对象
 * @returns 编译结果
 * @throws SchemaCompileError 编译失败时（类型未知、循环引用、语法错误等）
 */
function compileSchema(schemaData: Record<string, any>): CompiledSchema;

/**
 * 使用已编译的 Schema 校验变量状态
 * @param schema 编译后的 Schema
 * @param data 当前变量状态
 * @param context 运行时上下文（提供 refer() 所需的变量值查询能力）
 * @returns 校验结果
 */
function validateWithSchema(
  schema: CompiledSchema,
  data: Record<string, any>,
  context: ValidationContext
): ValidationResult;

interface ValidationContext {
  /** 根据路径获取当前变量状态中的值（供 refer() 使用） */
  resolveRef: (path: string) => any;
}

interface ValidationResult {
  success: boolean;
  /** 校验失败时的错误详情列表 */
  errors: Array<{
    path: string;     // 失败字段的路径
    message: string;  // 人类可读的错误描述
    expected: string; // 期望的约束
    received: any;    // 实际收到的值
  }>;
}
```

### 编译流程

```text
Schema JSON 对象
    │
    ▼
[第一遍：提取 $defs]
    │  遍历顶层，识别 $defs 键
    │  为每个结构体递归编译为独立的 Zod 对象
    │  存入 defRegistry（名称 → Zod 对象的映射表）
    │
    ▼
[第二遍：编译主体变量树]
    │  递归遍历 Schema 树（跳过 $defs 本身）
    │  对每个节点：
    │    ├─ 以 $ 开头 → 约束规则，调用对应的约束处理器
    │    └─ 不以 $ 开头 → 变量路径节点，递归编译子树
    │
    ▼
[生成最终 Zod 对象]
    │  组合所有节点的 Zod 定义
    │  处理 refer() 引用（生成延迟求值函数）
    │
    ▼
CompiledSchema
```

### 类型映射规则

| Schema 声明 | Zod 映射 | 说明 |
|-------------|----------|------|
| `$type: number` | `z.number()` | 标准数值 |
| `$type: number(force)` | `z.union([z.number(), z.string().pipe(z.coerce.number())])` | 接受字符串形式的数值并自动转换 |
| `$type: integer` | `z.number().int()` | 整数 |
| `$type: integer(force)` | 同 number(force) + `.int()` | 容错整数 |
| `$type: string` | `z.string()` | 标准字符串 |
| `$type: string(force)` | `z.coerce.string()` | 接受数值输入并转为字符串 |
| `$type: boolean` | `z.boolean()` | 布尔值 |
| `$type: any` | `z.any()` | 不校验 |
| `$type: object` | `z.object({...}).strict()` | 固定结构，默认不允许额外字段 |
| `$type: object` + `$extensible: true` | `z.object({...}).passthrough()` | 允许额外字段 |
| `$type: "array<T>"` | `z.array(compile(T))` | 数组，T 可以是基础类型名或 $defs 结构体名 |
| `$type: "record<T>"` | `z.record(z.string(), compile(T))` | 字典 |
| `$type: [A, B]` | `z.union([compile(A), compile(B)])` | 联合类型 |

### 约束处理规则

| 约束关键字 | 处理方式 |
|-----------|---------|
| `$min` / `$max` | 值为字面量时：`.min(n)` / `.max(n)`；值为 `refer(path)` 时：生成 `.refine()` 延迟校验 |
| `$enum` | 遍历值列表：字面量值用 `.literal()` 匹配，含通配符的值用 `.refine()` + 通配符匹配函数 |
| `$optional` | 将字段标记为 `.optional()` |
| `$default` | 添加 `.default(value)` |
| `$minLength` / `$maxLength` | `.min(n)` / `.max(n)`（字符串长度） |
| `$regex` | `.regex(new RegExp(pattern))` |
| `$minItems` / `$maxItems` | `.min(n)` / `.max(n)`（数组/record 长度），`refer()` 时同样延迟求值 |
| `$key_rule` | 对 record 的键施加约束：`.refine()` 校验所有键名 |
| `$extensible` | `true` 时使用 `.passthrough()`，`false` 时使用 `.strict()` |
| `$either` | 将数组中每组约束各自编译为 Zod 对象，用 `z.union()` 组合 |
| `$!` 前缀 | 编译原始约束后，用 `.refine(v => !originalCheck(v))` 取反 |

### `$defs` 解析

1. 提取 Schema 顶层的 `$defs` 对象。
2. 对 `$defs` 中的每个结构体，递归调用编译流程，生成独立的 Zod 对象。
3. 将结果存入 `defRegistry: Map<string, z.ZodType>`。
4. 后续编译中遇到 `"array<结构体名>"` 或 `"record<结构体名>"` 时，从 `defRegistry` 中查找对应的 Zod 对象。
5. 若引用的结构体名不存在于 `defRegistry` 且不是基础类型名，抛出 `SchemaCompileError`。

### `refer()` 处理

`refer(变量路径)` 在约束参数中引用其他变量的运行时值。编译器不在编译时求值，而是生成**延迟校验函数**：

```typescript
// 编译 $max: refer(角色/HP上限) 的处理逻辑
if (isRefer(constraintValue)) {
  const refPath = extractReferPath(constraintValue); // "角色/HP上限"
  // 不直接调用 .max(n)，改为 .refine() 延迟求值
  zodType = zodType.refine(
    (val) => {
      const refVal = context.resolveRef(refPath);
      return typeof refVal === 'number' ? val <= refVal : true;
    },
    (val) => ({
      message: `值 ${val} 超过了 refer(${refPath}) 的当前值`
    })
  );
}
```

`refer()` 引用不存在的路径时，对应约束**静默跳过**（不报错，视为无约束），避免变量初始化顺序导致的误判。

### 错误处理

编译错误不广播重试事件（这是角色卡配置问题，非 AI 生成问题）。编译失败时：

- 收集所有错误（不因第一个错误就中断），一次性输出完整的错误列表。
- 每条错误包含：出错位置（Schema 中的路径）、错误类型、人类可读的描述。
- 通过通知系统以 error 级别输出。

## 四、模块 3：JSON Patch 引擎

> 支撑功能：Update 变量更新（五·D-1 ~ D-4）、校验与错误处理（七·F-1 ~ F-2）

### 职责

解析并执行 `<Var_Update>` 标签中的更新指令（replace / insert / delete）。该模块的设计目标是**最大程度容忍 AI 模型输出的格式偏差**，通过多层预处理和模糊路径匹配，在保证数据安全的前提下尽可能让指令成功执行。

### 接口

```typescript
/**
 * 解析并执行一组更新指令
 * @param rawText <Var_Update> 标签内的原始文本
 * @param currentData 当前变量状态的深拷贝
 * @param schema 已编译的 Schema（用于逐条校验）
 * @param context 校验上下文（供 refer() 使用）
 * @returns 执行结果
 */
function executeUpdate(
  rawText: string,
  currentData: Record<string, any>,
  schema: CompiledSchema,
  context: ValidationContext
): UpdateResult;

interface UpdateResult {
  /** 执行后的变量状态 */
  data: Record<string, any>;
  /** 成功执行的指令数 */
  appliedCount: number;
  /** 被丢弃的指令（含失败原因） */
  discarded: Array<{ instruction: PatchInstruction; reason: string }>;
  /** 变更日志（路径 → "旧值 → 新值"） */
  log: Record<string, string>;
}
```

### 架构：三层管道

```text
原始文本（<Var_Update> 标签内容）
    │
    ▼
┌─────────────────────┐
│  第一层：指令预处理   │  文本清洗 → 宽松解析 → 语义规范化
└──────────┬──────────┘
           │ PatchInstruction[]
           ▼
┌─────────────────────┐
│  第二层：路径解析     │  反向模糊匹配 → 路径消歧义 → 确定唯一变量位置
└──────────┬──────────┘
           │ 路径已确定的 PatchInstruction[]
           ▼
┌─────────────────────┐

└─────────────────────┘
```

### 第一层：指令预处理管道

将 AI 模型的原始输出转换为合法的指令数组。按以下固定顺序依次执行：

#### 步骤 1：文本清洗

- **Markdown 围栏**：trim 后的整段若符合多行围栏（首行整行 `` ``` `` 或 `` ```lang `` 且该行无其它反引号，末行整行 `` ``` ``）或单行紧凑围栏（以 `` ``` `` 起、以末尾 `` ``` `` 止），则剥除一层；至多 5 次。围栏识别仅用行界与整段头尾，与 JSON 解析无关。
- 去除 BOM 标记。
- **截取指令 JSON**：第一个 `[` 与最后一个 `]`（若无则第一个 `{` 与最后一个 `}`）之间的子串，作为后续解析输入；去掉围栏外与数组/对象外的前后缀文字。

#### 步骤 2：引号修正（外向内对称解析）

AI 模型在输出 JSON 时最常见的格式错误集中在引号处理上。本步骤利用 JSON 结构和自然语言标点的**对称性**，从外部确定真正的字符串边界，统一解决两类问题。

**核心原理——外向内对称解析**：

JSON 的语法符号（`{` `}` `[` `]` `"` `:` `,`）和字符串值内容中的符号，都具有对称性。利用这一特性，不从前向后逐字符解析（容易被值内的 `"` 打断），而是**从外向内双向定位**字符串的真正边界。

```text
示例：{"value": "今天是"进化日"，皮尔特沃夫最重要的节日"}

传统正向解析会在此处断裂：
  {"value": "今天是"  ← 误以为字符串在此结束

外向内解析：
  从左侧：找到 : 之后的第一个 " → JSON 语法的字符串开始
  从右侧：找到 } 之前的最后一个 "  → JSON 语法的字符串结束
  
  左界                                                 右界
    ↓                                                    ↓
    "今天是"进化日"，皮尔特沃夫最重要的节日"
           ↑      ↑
           值内容的一部分 → 自动补 \" 转义
```

**同步解决的两个问题**：

**问题A：未转义的双引号（解析错误）**

- 典型场景：Claude 系列模型无法输出中文引号（“”），在 JSON 值中直接使用 ASCII 双引号 `"`，导致 JSON 解析器误判字符串边界。
- 修正策略：通过外向内解析确定真正的字符串起止位置后，将边界内的所有裸 `"` 自动补 `\"` 转义。

**问题B：被迫降级的单引号（内容失真）**

- 典型场景：模型为避免引号冲突，主动将值中的双引号降级为单引号（`'...'`），虽然不会导致解析错误，但不符合自然语言标点规范。
- 修正策略：确定字符串边界后，扫描值内容中的 `‘...’` 包裹模式，将其升级为双引号（`‘你好’` → `“你好”`）。此修正在解析成功后、Schema 校验前执行。

#### 步骤 3：宽松 JSON 解析

使用 json5 库解析，容忍以下 JSON 格式偏差：
- 尾逗号：`[{...},]`
- 无引号键名：`{op: "replace"}`
- 单引号字符串：`{'op': 'replace'}`（作为 JSON 语法的单引号，非步骤 2 处理的值内单引号）
- 注释：`// ...` 或 `/* ... */`

若 json5 也无法解析，判定为解析失败。说明：JSON 括号未闭合等「正文截断」通常伴随 `</Var_Update>` 缺失，模块 5 会先判定标签截断；主控制器按面向用户 G-1 略过该段标签解析，本条不再进入 Patch 管道。

#### 步骤 4：语义规范化

- 将 `"op": "add"` 映射为 `"op": "insert"`（AI 可能使用 RFC 6902 标准术语）。
- 将 `"op": "remove"` 映射为 `"op": "delete"`。
- 校验每条指令的结构：必须包含 `op` 和 `path` 字段，`replace` 和 `insert` 必须包含 `value` 字段。
- 不合法的指令条目直接丢弃，不阻断后续条目的处理。

### 第二层：反向路径解析

传统的路径解析是正向匹配——从根节点出发逐层向下查找，路径中任何一层不存在即报错。对深层嵌套结构，AI 模型极易在中间层级犯错（漏写一级、写错父节点名称等）。

本模块采用**反向路径解析**策略：

```text
指令路径：/角色/状态效果/中毒/持续回合
                              ▲
                              │ 从叶子节点开始
                              
第一步：提取叶子键名 "持续回合"
    → 在当前变量状态中全局搜索所有名为 "持续回合" 的字段
    → 找到候选集：[
        "角色/状态效果/中毒/持续回合",
        "敌人/状态效果/灼烧/持续回合"
      ]

第二步：向上逐级消歧义
    → 路径的倒数第二段是 "中毒"
    → 过滤候选集：仅保留父节点为 "中毒" 的
    → 剩余：["角色/状态效果/中毒/持续回合"]
    → 唯一匹配，确定目标路径

第三步：路径修正
    → 若指令原路径与匹配路径不一致，自动修正
    → 记录修正日志（通知系统 **trace**，仅控制台、需面板为 **debug** 等级）
```

**消歧义失败的处理**：

- 候选集在逐级过滤后仍有**多个**匹配 → 路径歧义，丢弃该条指令。
- 候选集在第一步就为**空**（叶子键名不存在）→ 对于 `replace` 和 `delete` 视为路径错误并丢弃；对于 `insert` 视为正常新增。

**数组索引的特殊处理**：

- 路径中的数字段（如 `/背包/0/名称`）：数字段视为数组索引，不参与反向搜索的键名匹配，但参与消歧义阶段的验证。
- 路径末端的 `-`（如 `/背包/-`）：语义为"数组末尾追加"，直接转发给执行层处理。

### 第三层：执行与校验

#### 逐条执行流程

```text
for instruction in instructions:
    1. 保存当前变量树根对象的深拷贝快照（用于单条失败时回滚）
    2. 执行操作：
       - replace: 检查路径存在 → 替换值
       - insert:  检查路径不存在（或为数组追加） → 写入值
       - delete:  检查路径存在 → 删除
    3. 用 Schema 校验整体变量状态
    4. 校验通过 → 记录变更日志，继续
       校验失败 → 回滚到步骤 1 的快照，记录丢弃原因，继续下一条
```

#### 多段合并

同一消息中多段 `<Var_Update>` 的指令数组合并为一个，按出现顺序排列。若同一路径出现多条指令，仅保留**最后一条**，之前的同路径指令丢弃，通过通知系统输出 **warning**（场景 `pat`；面板为 **notice** 等级时仍会 toastr）。

#### 变更日志

每条成功执行的指令生成一条日志记录，格式为 `"旧值 → 新值"`，写入当前楼层的 `log` 字段。

| `chat` | `getVariables({type: 'chat'})` / `replaceVariables(data, {type: 'chat'})` | `chat_metadata.variables` |

- 读写操作前自动执行深拷贝（通过 `klona`），避免外部意外修改持久化数据。

## 六、模块 5：消息标签提取器

> 支撑功能：标签识别（八·G-1）

### 职责

从消息文本中识别并提取 `<Var_Initial>` / `<Var_Update>` 标签及其内容。采用**分步检测**：先定位开标签，再校验闭标签；若缺闭标签则标记 `truncated`，供主控制器按面向用户 G-1 处理。

### 接口

```typescript
interface ExtractedTag {
  type: 'initial' | 'update';
  content: string;      // 标签内的原始文本内容
  startIndex: number;    // 标签在消息中的起始位置
  endIndex: number;      // 标签在消息中的结束位置
}

interface ExtractionResult {
  tags: ExtractedTag[];
  /** 是否检测到截断（有开标签但无对应闭标签） */
  truncated: boolean;
  /** 截断的标签类型（用于错误报告） */
  truncatedType?: 'initial' | 'update';
}

/**
 * 从消息文本中提取所有变量标签
 * @param messageText 完整消息正文（须来自 `chat[].mes` 等原始数据，勿使用 DOM 渲染结果）
 * @returns 提取结果
 */
function extractVarTags(messageText: string): ExtractionResult;
```

### 分步检测流程

```text
第一步：扫描开标签
  → 在消息文本中搜索所有匹配的开标签位置
  → 若无任何开标签 → 返回空结果（此消息不含变量指令）

第二步：逐个检查闭标签
  → 对每个检测到的开标签，搜索对应的闭标签
  → 闭标签存在 → 提取中间内容，加入结果列表
  → 闭标签不存在 → 判定为截断，设置 truncated = true

第三步：返回结果
  → 完整闭合的标签正常返回
  → truncated 与 truncatedType 一并返回，由主控制器决定是否继承上一层 data 等（见面向用户 G-1）
```

### 宽松标签名匹配

AI 模型可能输出多种标签命名变体。标签匹配采用**大小写不敏感 + 多格式兼容**策略：

以下变体均视为等价的 `Var_Update` 开标签：

| 变体 | 说明 |
|------|------|
| `<Var_Update>` | 标准格式 |
| `<var_update>` | 全小写 |
| `<VAR_UPDATE>` | 全大写 |
| `<VarUpdate>` | 驼峰无下划线 |
| `<varupdate>` | 全小写无下划线 |
| `<variable_update>` | 未缩写 |
| `<VariableUpdate>` | 驼峰未缩写 |

`Var_Initial` 同理（initial / Initial / INITIAL / Var_Initial / VarInitial / variable_initial 等）。

实现方式：将标签名标准化后匹配，而非枚举所有变体。标准化规则：转小写 → 移除下划线 → 与 `varupdate` / `variableupdate` / `varinitial` / `variableinitial` 比较。

闭标签的名称必须与对应开标签的标准化结果一致（但格式可以不同，如 `<Var_Update>...</var_update>` 视为合法闭合）。

### 转义支持

用户在消息中引用标签名（而非作为实际指令）时，可通过转义避免被识别：

| 转义方式 | 示例 | 说明 |
|----------|------|------|
| 反斜杠 | `\<Var_Update>` | 反斜杠后的标签不被识别 |
| 代码块 | `` `<Var_Update>` `` | 行内代码块中的标签不被识别 |

扫描时跳过以上两种转义形式，不将其纳入标签匹配候选。

### 其他规则

- 同一消息中允许出现多个同类标签，各自独立提取。
- 提取后的 `content` 保留原始文本（包括首尾空白），由下游模块（格式解析器或 JSON Patch 引擎）处理。
- 数据源：始终从 `chat[messageIndex].mes` 读取原始消息文本，不从 DOM 渲染后的文本读取（避免显示正则污染）。

## 七、模块 6：插值宏引擎

> 支撑功能：插值宏格式与输出（十二·K-1 ~ K-4）

### 职责

注册自定义插值宏，在消息发送前将 `{{作用域/字段/路径}}` 格式的宏替换为变量的实际值。复合类型值使用 PromptalYAML 格式输出。

### 接口

```typescript
/**
 * 注册所有 VarUpdate 插值宏
 * @returns 注销函数（脚本卸载时调用）
 */
function registerMacros(): () => void;
```

### 宏格式

统一使用 `{{作用域/字段/变量路径}}` 格式：

- `作用域`：`message` | `chat` | `global`
- `字段`：`data`（变量数据）| `log`（变更日志）| `schema`（Schema 原始数据）
- `变量路径`：`/` 分隔的任意深度路径，可省略（省略时返回该字段下的完整对象）

注册的正则表达式：`/\{\{(message|chat|global)\/(data|log|schema)(?:\/([^}]*))?\}\}/g`

### 替换逻辑

```text
1. 从正则匹配结果中提取 scope、field、path
2. 调用模块 4（变量存储）读取对应的值
3. 判断值的类型：
   ├─ undefined/null → 替换为空字符串，通知系统输出 notice
   ├─ 基础类型（number/string/boolean）→ 直接转为字符串
   └─ 复合类型（object/array）→ 调用 PromptalYAML 序列化器
4. 执行缩进对齐（见下文）
5. 返回替换后的字符串
```

### PromptalYAML 序列化器

将 JavaScript 对象/数组序列化为 PromptalYAML 文本，实现于 `shared/promptal-yaml.ts`。

#### 序列化规则

```typescript
function serializeToPromptalYAML(value: any, indentLevel?: number): string;
```

**键值对**：
- 输出 `键名: 值`，键名不加引号。
- 嵌套层级每增加一级，缩进增加两个空格。
- 字符串值不加引号，直接输出原始内容。

**数组**：
- 元素全部为基础类型时，使用内联格式：`[元素1, 元素2, 元素3]`
- 存在复合类型元素时，使用展开格式：

```text
- 名称: 铁剑
  品质: 精良
- 名称: 皮甲
  品质: 普通
```

**空集合**：空数组 → `[]`，空对象 → `{}`

**多行字符串**：
- 检测条件：字符串值包含换行符（`\n`）。
- 输出格式：

```text
键名:
  ```
  第一行内容
  第二行内容
  ```
```

- 开闭反引号行与内容行保持相同的缩进层级。
- 块内内容原样输出，不做额外缩进处理。

**布尔值和数值**：直接输出 `true`/`false` 和数字字面量。

### 缩进对齐

宏替换时，自动检测宏所在行前方的空格数 `N`，将输出的每一行（除第一行外）左侧补齐 `N` 个空格：

```typescript
function alignIndent(output: string, leadingSpaces: number): string {
  const lines = output.split('\n');
  return lines
    .map((line, i) => i === 0 ? line : ' '.repeat(leadingSpaces) + line)
    .join('\n');
}
```

## 八、模块 7：事件总线

> 支撑功能：脚本间通信（八·G-2）

### 职责

封装酒馆助手事件 API，定义 VarUpdate 与外部脚本（如 Agents）之间通信的事件协议。

### 事件定义

##### VarUpdate 广播的事件

| 事件名 | 触发时机 | 说明 |
|--------|---------|------|
| `varupdate:initialized` | 变量初始化完成 | 通知其他脚本变量已就绪 |
| `varupdate:updated` | 变量更新成功 | **Agents 重试联动的成功信号**。携带完整变更日志 |
| `varupdate:update_failed` | 变量更新失败（丢弃指令数超过容错阈值或整段解析失败） | **Agents 重试联动的失败信号**。触发 Agents 广播 `varupdate:retry_requested` |
| `varupdate:schema_ready` | Schema 编译完成 | 通知 Schema 状态就绪 |

##### VarUpdate 监听的事件

| 事件名 | 来源 | 用途 |
|--------|------|------|
| `agents:message_complete` | Agents | **主消息通道**。从负载 `content` 读取消息文本进行标签提取 |
| `varupdate:retry_requested` | Agents | 回退指定消息的变量状态，准备重新接收 |

##### 事件负载类型定义

```typescript
/** varupdate:initialized */
interface VarInitializedPayload {
  messageIndex: number;      // 初始化数据写入的消息索引
  data: Record<string, any>; // 初始化后的完整变量状态
}

/** varupdate:updated */
interface VarUpdatedPayload {
  messageIndex: number;      // 触发更新的消息索引
  appliedCount: number;      // 成功执行的指令数
  discardedCount: number;    // 被丢弃的指令数
  log: Record<string, string>; // 变更日志（路径 → "旧值 → 新值"）
}

/** varupdate:update_failed */
interface VarUpdateFailedPayload {
  messageIndex: number;      // 触发更新的消息索引
  reason: string;            // 失败原因描述
  discardedCount: number;    // 被丢弃的指令数
}

/** varupdate:schema_ready */
interface SchemaReadyPayload {
  defNames: string[];        // $defs 中定义的结构体名称列表
}

/** varupdate:retry_requested（由 Agents 广播，VarUpdate 监听） */
interface RetryRequestedPayload {
  messageIndex: number;      // 需要回退变量的消息索引
}

/** agents:message_complete（由 Agents 广播，VarUpdate 监听） */
interface MessageCompletePayload {
  agentId: string;           // 智能体唯一 ID
  agentLabel: string;        // 智能体标签名
  stepIndex: number;         // 步骤序号
  content: string;           // 生成的完整消息文本（经后处理）
  reasoning?: string;        // 推理/思维链内容
  messageIndex?: number;     // chat[] 消息索引（「仅解析」模式为 undefined）
  writeMode?: string;      // 写入方式（建议填充；VarUpdate 不据此分支，可省略）
}
```

### 接口

```typescript
/**
 * 广播事件
 */
function emit(eventName: string, payload: any): void;

/**
 * 监听事件
 * @returns 包含 stop 方法的对象（与酒馆助手 eventOn 返回值一致）
 */
function on(eventName: string, handler: (payload: any) => void): { stop: () => void };

/**
 * 一次性监听
 * @returns 包含 stop 方法的对象（与酒馆助手 eventOnce 返回值一致）
 */
function once(eventName: string, handler: (payload: any) => void): { stop: () => void };

// 注销监听：使用 on() / once() 返回的 { stop }。

/**
 * 注销所有由本脚本注册的事件监听器
 * 在脚本卸载时调用
 */
function removeAll(): void;
```

### 实现

直接委托酒馆助手的 `eventEmit` / `eventOn` / `eventOnce` / `eventClearAll`。事件名使用 `varupdate:` 前缀命名空间，避免与其他脚本冲突。

## 九、模块 8：通知系统

> 支撑功能：通知提示（十一·J-2 ~ J-3）

### 职责

统一的五级通知输出，根据用户设置的通知等级控制 toastr 弹窗和控制台的可见性。

### 通知等级

| 等级 | 数值 | toastr | 受等级过滤的控制台 | 说明 |
|------|------|--------|-------------------|------|
| `debug` | 0 | info+成功+警告+错误 | 同上 + **trace**（仅控制台） | `trace()` 不弹 toastr，防刷屏 |
| `always` | 1 | 成功+警告+错误 | 成功+警告+错误 | — |
| `notice` | 2 | 警告+错误 | 警告+错误 | 默认 |
| `error` | 3 | 仅错误 | 仅错误 | — |
| `silence` | 4 | 无常规通知；无 `feedback` | 无常规日志与 trace、无快照 | **`bridgeError` / `bootstrapError` 仍 `console.error`** |

**场景码**（控制台与 toastr 标题）：`boot` `evt` `msg` `sch` `pat` `wb` `ui` `mac` `man` `life` `state` `sys`。前缀形如 `[VarUpdate][pat]`。

### 接口

```typescript
type NotifyCategory =
  | 'boot' | 'evt' | 'msg' | 'sch' | 'pat' | 'wb' | 'ui' | 'mac' | 'man' | 'life' | 'state' | 'sys';
type NotifyOptions = { category?: NotifyCategory };

/**
 * 设置当前通知等级（数值大于消息所需等级则忽略该条）
 */
function setLevel(level: NotifyLevel): void;

function getLevel(): NotifyLevel;

function notify(level: NotifyLevel, title: string, message?: string, opts?: NotifyOptions): void;

/** toastr info + 控制台（仅 debug 等级）；opts.category 控制场景码 */
function debug(title: string, message?: string, opts?: NotifyOptions): void;
function success(title: string, message?: string, opts?: NotifyOptions): void;
function warning(title: string, message?: string, opts?: NotifyOptions): void;
function error(title: string, message?: string, opts?: NotifyOptions): void;

/** 仅控制台、不 toastr；仅 debug 等级 */
function trace(title: string, message?: string, category?: NotifyCategory): void;

/** silence 以外不受等级过滤；默认 category 为 man */
function feedback(isSuccess: boolean, title: string, message?: string, opts?: NotifyOptions): void;

/** 总线桥接失败：始终 console.error；toastr 在非 silence 且等级允许 error 时 */
function bridgeError(context: string, err: unknown): void;

/** init 顶层失败：始终 console.error；非 silence 时 toastr */
function bootstrapError(err: unknown): void;

/** 变量快照：debug 打全量；always 一行摘要；更高等级不打 */
function logStateSnapshot(data: Record<string, any>): void;
```

### 实现

## 十一、模块 10：条件求值引擎

> 支撑功能：变量条件标签（十三·L-1 ~ L-14）

### 职责

解析 `[]` 变量条件标签内的表达式文本，读取当前变量状态，求值返回布尔结果。本模块为纯逻辑模块，不关心标签来自哪类条目（世界书/预设/正则），仅负责"给一个表达式字符串和一份变量数据，返回 true/false"。

### 接口

```typescript
/**
 * 对单个条件表达式求值
 * @param expression 表达式文本（不含定界符 []），如 "HP" >= 60
 * @param data 当前变量快照（最新一层 message.data）
 * @returns 求值结果；解析失败或变量不存在时返回 false
 */
function evaluateCondition(expression: string, data: Record<string, any>): boolean;

/**
 * 对条目标题/备注中的所有变量条件标签求值
 * 提取所有 [] 标签，逐个求值，多标签取 AND。
 * @param text 含标签的文本（如 "[\"HP\" >= 60][\"MP\" > 0] 战斗设定"）
 * @param data 当前变量快照
 * @returns 所有标签均通过返回 true；无标签时返回 true（不过滤）
 */
function evaluateAllConditions(text: string, data: Record<string, any>): boolean;
```

### 表达式解析

表达式结构为 `[可选!] 左操作数 运算符 [右操作数]`，解析流程：

1. **顶层分割**：按 `|` 分割为多个 OR 分支（注意转义和引号内的 `|` 不分割）。
2. **NOT 检测**：去除首位的 `!` 前缀，标记取反。
3. **操作数与运算符提取**：
   - 左操作数：引号包裹的变量路径字符串（`"变量名"` 或 `"a/b/c"`）。
   - 运算符：`==` `!=` `===` `!==` `>` `>=` `<` `<=` `?` `!?` `∋` `!∋` `⊇` `!⊇` `#`。
   - 右操作数：字面量（数字、字符串、布尔值 `true`/`false`/`null`）或变量引用（`$"变量名"`）。
4. **存在性检查**（`?` / `!?`）无右操作数。
5. **长度运算符**（`#`）后接比较运算符和数值。

### 求值逻辑

| 运算符 | 求值方式 |
|--------|---------|
| `==` / `!=` | 宽松相等：类型不同时尝试 Number/String 转换后比较 |
| `===` / `!==` | 严格相等：`Object.is()` 语义 |
| `>` `>=` `<` `<=` | 按 `Number()` 转换后数值比较；转换失败返回 false |
| `?` / `!?` | `data` 中路径对应值是否为 `undefined` 或 `null` |
| `∋` / `!∋` | `Array.isArray(val)` 后遍历元素，字符串元素支持通配符匹配 |
| `⊇` / `!⊇` | `typeof val === 'object'` 后检查 `Object.keys()`，键名支持通配符匹配 |
| `#` + 比较 | 取 `Array.length` 或 `Object.keys().length`，再执行数值比较 |

**变量引用**（`$"变量名"`）：右操作数为变量引用时，先从 `data` 读取其值，再参与比较。

**通配符匹配**：所有涉及字符串值比较的场景，若字面量或引用值含 `*`，使用 `wildcardMatch()` 进行模糊匹配。规则与 Schema `$enum` 通配符一致。

### 错误处理

- 表达式语法错误：返回 `false`，输出 `WARN` 级日志（受通知等级控制）。
- 变量路径不存在：返回 `false`，不输出日志（静默不匹配）。
- 变量数据为空（`data` 为 `null`/`undefined`）：所有表达式返回 `false`。
- 单个 OR 分支求值失败不影响其他分支。

## 十二、模块 11：原生过滤 Hook

> 支撑功能：变量条件标签的作用范围与判定流程（十三·L-11 ~ L-12）

### 职责

在酒馆生成请求前，Hook 世界书激活、预设注入和正则脚本执行的流程，对携带变量条件标签的条目调用模块 10 进行求值过滤。未通过的条目从本次请求中排除。

### 接口

```typescript
/**
 * 注册所有过滤 Hook
 * 在主控制器初始化时调用
 */
function registerFilterHooks(): void;

/**
 * 注销所有过滤 Hook
 * 在脚本卸载时调用
 */
function unregisterFilterHooks(): void;
```

### 过滤目标

| 条目类型 | 标签写入字段 | Hook 点 |
|----------|-------------|--------|
| 世界书条目 | `comment`（备注） | `WORLDINFO_SCAN_DONE` 事件回调 |
| 预设条目 | `name`（标题） | `SENDING_MESSAGE` 事件回调 |
| 正则脚本 | `scriptName`（名称） | `SENDING_MESSAGE` 事件回调 |

### 过滤流程

```text
Hook 触发
  → 从模块 4（变量存储）读取最新一层 message.data
  → 遍历候选条目列表：
     ├─ 提取该条目标识字段中的所有 [] 标签
     ├─ 无标签 → 跳过（自动通过）
     └─ 有标签 → 调用模块 10 的 evaluateAllConditions() 对全文求值
        ├─ 返回 true → 保留
        └─ 返回 false → 从候选列表中移除
  → 返回过滤后的候选列表
```

### 世界书条目过滤

监听 `WORLDINFO_SCAN_DONE` 事件。该事件在酒馆完成世界书关键词扫描后、实际注入提示词前触发，回调参数中包含已激活的条目列表。过滤器遍历列表，读取每个条目的 `comment` 字段，解析其中的变量条件标签，未通过的条目从列表中移除。

### 预设条目过滤

监听 `SENDING_MESSAGE` 事件。该事件在酒馆组装最终发送给 API 的消息列表时触发。过滤器检查消息列表中来自预设的条目，读取其 `name` 字段进行条件标签求值，未通过的条目从列表中排除。

### 正则脚本过滤

正则脚本的过滤与预设条目在同一 Hook 点进行。酒馆在消息发送前执行正则脚本列表，过滤器检查每个脚本的 `scriptName` 字段，未通过条件的脚本跳过执行。

### 与 Agents 的协作

两层过滤独立执行、结果取 AND：
- Agents 脚本通过 `【】` 智能体标签决定条目是否参与某个步骤。
- VarUpdate 通过 `[]` 变量条件标签决定条目是否满足当前变量状态。
- 两者都通过的条目才最终注入。
- 两个过滤器互不感知对方的存在，各自独立运行。

## 十四、主控制器概述

主控制器（`index.ts`）是脚本的入口点和业务流程编排者，不属于基础设施模块，但负责调度所有模块协同工作。主要流程：

### 初始化流程

```text
脚本加载
  → 模块 9（UI 面板）渲染设置面板
  → 模块 6（插值宏）注册宏
  → 模块 7（事件总线）注册酒馆事件监听
  → 模块 11（原生过滤）注册世界书/预设/正则过滤 Hook
  → 等待聊天加载完成
```

### 消息生成后的处理流程

双通道、`messageIndex` 校验与 `notify.debug` / `writeMode` 的 trace 等**以《接口与契约集中定义》「VarUpdate 双通道消息处理」为准**；此处仅保留汇合后的数据流。

```text
事件触发（两个独立通道 + 宿主编辑/滑动等，数据源与入口见契约）：

  汇合后统一：
  → 模块 5（标签提取）分步检测：
     ├─ 检测到开标签但无闭标签 → 标记截断；主控制器略过标签解析，仍执行继承（见面向用户 G-1）
     └─ 检测到完整标签对 → 提取内容，继续
  → 若存在 <Var_Initial>:
       → 模块 1（格式解析）解析标签内容
       → 模块 4（变量存储）写入 message 层
  → 若存在 <Var_Update>:
       → 模块 3（JSON Patch）执行更新、校验
       → 模块 4（变量存储）写入更新后的状态
       → 模块 7（事件总线）广播 varupdate:updated 或 varupdate:update_failed
  → 模块 8（通知系统）输出处理结果
```

### 需要监听的触发事件

以下事件可能导致变量状态发生变化，需要重新读取或回退 schema / default / initial / update：

| 触发场景 | 对应事件 | VarUpdate 的响应 |
|----------|----------|-----------------|
| AI 消息生成完成 | Agents 事件（主）/ `MESSAGE_RECEIVED`（备） | 提取标签、执行初始化或更新 |
| 新建聊天 | `CHAT_CHANGED` | 重新加载 Schema、重置变量状态 |
| 切换开场白（滑动首条消息） | `MESSAGE_SWIPED` | 回退到该分支的变量快照 |
| 用户手动编辑消息 | `MESSAGE_EDITED` | 重新提取编辑后消息的标签、重新执行 |
| 删除消息 / 回退到历史分支 | `MESSAGE_DELETED` | 负载为删除后的 `chat.length`（SillyTavern 约定）；清扫该下标起可能残留的 message 变量槽，并触发自动恢复变量链 |

### Schema 加载流程

```text
监听 CHAT_CHANGED 事件
  → 模块 4（变量存储）从 chat 层读取 Schema 原始数据
  → 模块 1（格式解析）解析
  → 模块 2（Schema 编译）编译为 Zod 对象
  → 模块 7（事件总线）广播 varupdate:schema_ready
```

### 脚本卸载流程

```text
监听 pagehide 事件
  → 模块 11（原生过滤）注销所有过滤 Hook
  → 模块 6（插值宏）注销宏
  → 模块 7（事件总线）移除所有监听器
```
