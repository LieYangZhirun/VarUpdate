# 变量更新脚本（VarUpdate）——面向用户功能卡

> **运行环境**：本脚本运行于 SillyTavern 的「酒馆助手」浏览器扩展所提供的 iframe 沙箱中，为 JavaScript 脚本执行环境。  
> **核心功能**：面向角色卡作者与普通用户，提供结构化变量管理：在世界书中定义 Schema 与 Default，在消息中处理 Initial 与 Update，并将变量状态持久化至酒馆助手的多层变量存储。  
> **协同脚本**：同仓库中的「多智能体脚本（Agents）」负责多模型流水线编排；可将某一智能体步骤的输出交由本脚本解析。两脚本经事件总线松耦合，可单独启用，也可在同一聊天环境中配合使用（常见部署为二者同时启用）。

---

## 一、核心概念

### 概念总览

| 概念 | 定义位置 | 作用 | 数据格式 |
|------|----------|------|----------|
| **Schema（结构定义）** | 世界书条目 `[Var_Schema]` | 规定变量的类型、约束和结构 | 声明式 YAML/JSON/TOML |
| **Default（默认值）** | 世界书条目 `[Var_Default]` | 为变量提供安全兜底值 | YAML/JSON/TOML |
| **Initial（初始值）** | 消息内容 `<Var_Initial>...</Var_Initial>` | 设定某一层的完整变量状态 | YAML/JSON/TOML |
| **Update（更新指令）** | 消息内容 `<Var_Update>...</Var_Update>` | 在已有变量基础上增量修改 | JSON Patch |

### 数据格式

所有需要用户提供结构化数据的输入（Schema、Default、Initial）均支持三种书写格式：
- **JSON**：标准 JSON 格式。
- **YAML**：对非程序员更友好的人类可读格式。
- **TOML**：键值对风格的配置文件格式。

脚本内部统一使用 JSON 进行处理和存储。无论用户以何种格式书写，解析后均转换为 JSON 对象。

### 执行时序

```text
创建新聊天 / 用户手动「重新加载格式规则」
    → 扫描世界书，加载 [Var_Schema] 条目 → 解析并存入 chat 层
    → 扫描世界书，加载 [Var_Default] 条目 → 解析并存入 chat 层
    → 从 chat 层的 Schema 声明编译为运行时校验器（Zod 对象）

每条消息生成完成后
    → 解析消息内容
    → 若含 <Var_Initial>：清空当前层基线，用 Initial 赋值 → Default 补全缺字段 → Schema 整体校验 → 写入
    → 若含 <Var_Update>：在「当前层已有 data 或继承自上一层」的基础上逐条执行；每条后 Schema 校验，失败则单条回滚
       （同一条消息可同时含 Initial 与 Update：总是先 Initial 再 Update）
    → 若不含 Initial 且不含 Update：从前一层继承 data，本层 log 置空 → 写入（见 D-4、G-1）
    → 收尾：如按设置清理远期楼层的变量副本等（见第十章）
```

## 二、Schema 结构定义

### A-1 定义方式

角色卡作者在世界书中创建一个条目，在备注（comment）中包含 `[Var_Schema]` 标签，条目内容为声明式的变量结构定义。

> **条目可见性**：Schema 条目的内容是给脚本解析的，不应注入到 AI 的上下文中（白白浪费 Token）。建议将条目设为禁用（Disabled）——脚本直接读取世界书原始数据，不依赖条目是否启用。若配合 Agents 脚本使用且希望条目保持启用，则必须加上 `【None】` 标签（Agents 脚本的智能体标签，含义为在所有智能体步骤中均不发送），避免注入。

> **多条目合并**：世界书中可存在多条备注含 `[Var_Schema]` 或 `[Var_Default]` 的条目。脚本按世界书返回顺序逐条解析为对象后做**深度合并**。若在**同一路径**上出现不一致的定义（与已合并结果冲突），则**报错并中止**该次加载，不静默覆盖；作者需消除冲突后使用「重新加载格式规则」。

#### 约束标记符规则

Schema 中既包含变量本身的层级结构，也包含对变量的约束规则。为了区分两者，所有约束关键字均使用 **`$` 前缀**：

- 以 `$` 开头的键名 → 约束规则（由脚本解析，不作为实际变量名）。
- 不以 `$` 开头的键名 → 变量名或变量层级（构成实际的变量树结构）。

#### 综合示例

```yaml
# 世界书条目示例
# comment: [Var_Schema]

$defs:
  物品:
    $type: object
    名称: { $type: string }
    数量: { $type: integer, $min: 1 }
    品质: { $type: string, $optional: true, $enum: ["普通", "精良", "传说"] }
  技能:
    $type: object
    名称: { $type: string }
    冷却: { $type: integer, $min: 0 }
  派系:
    $type: object
    $extensible: true
    政体: { $type: string, $optional: true }
    描述: { $type: string, $optional: true }
    成员列表: { $type: "record<string>", $optional: true }

角色:
  $type: object
  HP:
    $type: number(force)
    $min: 0
    $max: refer(角色/HP上限)
  HP上限:
    $type: number(force)
    $default: 999
  MP:
    $type: number(force)
    $min: 0
  状态:
    $type: string
    $enum: ["正常", "中毒", "昏迷"]
  技能列表:
    $type: "array<技能>"
    $maxItems: 10

背包:
  $type: "array<物品>"
  $maxItems: refer(背包容量)

背包容量:
  $type: integer
  $default: 20

势力:
  $type: "record<派系>"
```

以上定义中：
- `角色`、`HP`、`背包`、`势力` 等不带 `$` 前缀的键是实际变量路径名。
- `$type`、`$min`、`$max`、`$enum`、`$defs` 等带 `$` 前缀的键是约束规则。
- `$defs` 中定义了 `物品`、`技能`、`派系` 三个可复用结构体。
- `array<物品>` 直接使用结构体名作为泛型参数。
- `refer(HP上限)` 和 `refer(背包容量)` 引用其他变量的运行时值作为约束参数。

### A-2 类型系统

#### 基础类型

| 类型名 | 含义 | Zod 映射 |
|--------|------|----------|
| `number` | 数值 | `z.number()` |
| `number(force)` | 容错数值，接受字符串输入并自动转换为数字 | `z.union([z.number(), z.string().pipe(z.coerce.number())])` |
| `integer` | 整数 | `z.number().int()` |
| `integer(force)` | 容错整数，同上 | 同上 + `.int()` |
| `string` | 字符串 | `z.string()` |
| `string(force)` | 容错字符串，接受数值输入并自动转换为字符串 | `z.coerce.string()` |
| `boolean` | 布尔值 | `z.boolean()` |
| `any` | 任意类型，不做类型校验 | `z.any()` |

> `force`：在常见输出偏差（如数字被写成字符串）下仍尽量通过校验并归一化为目标类型。

> **约定**：带 `(force)` 后缀的内置类型仅上表三种（`number` / `integer` / `string`），**没有** `boolean(force)`。详见《接口与契约集中定义》§4.4 与源码 `schema-force-primitives.ts`。

#### 容器类型

| 类型名 | 含义 | 语法 |
|--------|------|------|
| `object` | 固定结构对象 | `$type: object` |
| `array<T>` | 数组，元素类型为 T | `$type: "array<物品>"` 或 `$type: "array<string>"` |
| `record<T>` | 字典（键值对集合），键固定为 string，值类型为 T | `$type: "record<string>"` 或 `$type: "record<派系>"` |

- **`object`**：固定结构。声明的字段受约束校验，且**默认不允许添加未声明的字段**。若需要允许扩展，显式添加 `$extensible: true`。所有 object 类型节点均需显式声明 `$type: object`。
- **`array<T>`**：T 可以是基础类型名或 `$defs` 中定义的结构体名。数组可自由增删元素，但每个元素必须符合 T 的类型约束。
- **`record<T>`**：键固定为 string（无需声明），值必须符合 T。可自由增删键值对。若需要对键名施加额外约束（如正则或枚举），使用 `$key_rule`。

#### 联合类型（Union）

使用数组语法同时声明多个可接受的类型：

```yaml
灵活数值:
  $type: [number, string]   # 接受数字或字符串

混合列表:
  $type: ["array<string>", "array<number>"]  # 接受字符串数组或数字数组
```

#### 结构体定义与引用（`$defs`）

在 Schema 的**顶层**使用 `$defs` 定义可复用的结构体：

```yaml
$defs:
  通用对象:
    $type: object
    $extensible: true
    类型: { $type: string, $optional: true }
    描述: { $type: string, $optional: true }
    效果: { $type: string, $optional: true }

# 引用时直接使用结构体名作为类型名
仓储间:
  $type: "record<通用对象>"
战技:
  $type: "record<通用对象>"
```

`$defs` 必须位于 Schema 的**顶层**（与实际变量定义同级），不允许在嵌套层级中就地定义。`$defs` 本身不产生任何实际变量，仅作为结构体模板供引用。

#### 运行时值引用（`refer(xxx)`）

`refer(变量路径)` 用于在约束参数中引用其他变量的**运行时实际值**：

```yaml
角色:
  $type: object
  HP:
    $type: number
    $min: 0
    $max: refer(角色/HP上限)     # HP 的上限由同级变量 HP上限 的实际值决定
  HP上限:
    $type: number
    $default: 999

背包:
  $type: "array<物品>"
  $maxItems: refer(背包容量)  # 背包最大容量由变量决定

背包容量:
  $type: integer
  $default: 20
```

`refer` 仅可用于约束参数（如 `$min`、`$max`、`$maxItems` 等），不可用于 `$type`。路径使用 `/` 分隔符（与 Update 指令、插值宏共用同一套路径规范）。校验时脚本从当前变量状态中读取被引用变量的实际值，动态构建约束。

### A-3 约束关键字

#### 通用约束

| 关键字 | 适用类型 | 说明 | 示例 |
|--------|----------|------|------|
| `$type` | 所有 | 类型声明（见 A-2） | `$type: number` |
| `$optional` | 所有 | 标记为可选字段（默认必填） | `$optional: true` |
| `$default` | 所有 | 字段默认值 | `$default: 100` |
| `$enum` | 所有 | 允许的值列表，字符串值**支持通配符** | `$enum: [1, 2, 3]` 或 `$enum: ["***之王"]` |
| `$extensible` | object | 允许添加 Schema 中未声明的字段 | `$extensible: true` |
| `$hide` | 变量节点（与子字段定义同层的约束键） | `true` 时：**概括展开**（`{{message/data}}` 或父路径对象序列化）会整段省略该键及子树；**宏路径恰好终止于此节点**时替换为换行（空白行），不提示「引用为空」；**路径继续深入到该节点之下的子键**（如 `…/记忆系统/某叶子`）则**仍正常注入**该子键值。不影响校验与存储 | `记忆系统: { $hide: true, $type: object }` |

#### 数值约束

| 关键字 | 适用类型 | 说明 | 示例 |
|--------|----------|------|------|
| `$min` / `$max` | number, integer | 数值范围（含边界） | `$min: 0`, `$max: refer(HP上限)` |

> 通过 `$min`/`$max` 配合 `$either` 和 `$enum` 可组合实现各类数值比较逻辑。

#### 字符串约束

| 关键字 | 适用类型 | 说明 | 示例 |
|--------|----------|------|------|
| `$minLength` / `$maxLength` | string | 字符串长度限制 | `$minLength: 1` |
| `$regex` | string | 正则表达式匹配 | `$regex: "^[A-Z].*[0-9]$"` |

#### 容器约束

| 关键字 | 适用类型 | 说明 | 示例 |
|--------|----------|------|------|
| `$minItems` / `$maxItems` | array, record | 子项数量限制 | `$maxItems: refer(背包容量)` |
| `$key_rule` | record | 对字典键名施加额外约束 | `$key_rule: { $regex: "^NPC_" }` |

#### 通配符语法

通配符 `*` 可用于所有涉及字符串匹配的场景（`$enum`、`$key_rule` 等）：

**个数规则**：

- **1~2 个 `*`**：每个 `*` 严格匹配恰好 1 个字符。
- **3 个及以上 `*`**：匹配任意数量的字符（0 到无穷）。

**位置规则**：通配符在哪一侧，该侧就允许有额外内容；没有通配符的一侧必须是字符串的边界。

| 模式 | 含义 | 匹配 | 不匹配 |
|------|------|------|--------|
| `"***开心***"` | 任意位置包含"开心" | "很开心啊"、"开心" | — |
| `"***开心"` | 以"开心"结尾 | "很开心"、"开心" | "开心了" |
| `"开心***"` | 以"开心"开头 | "开心果"、"开心" | "很开心" |
| `"*开心*"` | 前后各恰好 1 字符 | "很开心啊" | "开心"、"超开心啊呀" |
| `"**民主主义"` | 前面恰好 2 字符 | "社会民主主义" | "民主主义"、"新民主主义" |

通配符和正则（`$regex`）同时可用，定位不同：

- **通配符**：简单场景，低门槛，可嵌入 `$enum` 中与精确值混用。
- **正则**：复杂场景，处理超长字符串+超复杂规则时使用。

### A-4 逻辑组合

#### 与（AND）

同一节点上的多个约束天然构成「与」关系，必须同时满足：

```yaml
HP:
  $type: number
  $min: 0
  $max: 999
  # → 必须是 number 且 >= 0 且 <= 999
```

#### 或（OR）——`$either`

`$either` 下辖一个**数组**，每个元素是一组独立的约束规则，满足其中任意一组即通过：

```yaml
状态值:
  $type: string
  $either:
    - $enum: ["正常", "中毒", "昏迷"]
    - $regex: "^自定义_"
  # → 必须是 string，且（是枚举值之一 或 匹配自定义前缀）
```

#### 非（NOT）——`$!` 前缀

在约束关键字的 `$` 后紧跟 `!` 表示取反：

```yaml
标签:
  $type: string
  $!enum: ["禁用A", "禁用B"]   # 不能是这些值
  $!regex: "^test_"             # 不能匹配此正则
```

### A-5 Schema 的内部处理

- **编译过程**：脚本读取 Schema 声明后，通过内置编译器将其转换为 Zod 校验对象。编译器递归遍历 Schema 树，将 `$` 前缀约束映射为 Zod 方法调用（如 `$type: number` → `z.number()`，`$min: 0` → `.min(0)`，`$extensible: true` → `.passthrough()`）。`$defs` 中的结构体先编译为独立的 Zod 对象，供后续引用。
- **存储分离**：Schema 的原始声明（纯 JSON 数据）存入 chat 层供持久化；编译后的 Zod 对象存在内存中供运行时校验使用。
- **变量管理器集成**：编译完成后，脚本将 Zod 对象通过酒馆助手提供的 `registerVariableSchema()` 接口注册给变量管理器。这使得用户在变量管理器中手动编辑变量时，能获得类型校验提示。

### A-6 Schema 的生命周期

- **加载时机**：创建新聊天时自动加载，或用户手动触发「重新加载格式规则」。
- **存储位置**：解析后的 Schema 声明（纯 JSON 数据）存入酒馆助手变量系统的 **chat 层**。
- **版本隔离**：Schema 一旦存入 chat 层，即与世界书解耦。后续角色卡作者更新世界书中的 Schema 定义不影响已有聊天的校验规则，仅对新创建的聊天生效。若需要将新规则同步到当前聊天，用户需手动触发「重新加载格式规则」。

## 三、Default 默认值

### B-1 定义方式

角色卡作者在世界书中创建一个条目，在备注（comment）中包含 `[Var_Default]` 标签，条目内容为变量的默认值定义，格式为 YAML、JSON 或 TOML。多条 `[Var_Default]` 条目的合并与冲突规则与 A-1「多条目合并」一致。

> **条目可见性**：与 Schema 条目相同，Default 条目也不应注入到 AI 上下文中。建议禁用或加标签屏蔽（详见 A-1）。

```yaml
# 世界书条目示例
# comment: [Var_Default]
# 内容：

角色:
  HP: 100
  MP: 50
  状态: "正常"
  技能列表: []
背包: []
```

### B-2 Default 的作用

Default 提供安全兜底值。在以下场景中生效：

- **Initial 填充后的补全**：若 Initial 中未包含 Schema 中声明的某个字段，脚本用 Default 中对应的值自动填充。
- **新建聊天时的初始状态**：若开场白中没有 `<Var_Initial>` 标签，脚本用 Default 值作为变量的初始状态。

> **`[Var_Default]` 与 `$default`**：`$default` 为 Schema 内联字段默认值；`[Var_Default]` 为世界书中的整体默认值。二者同时存在时，**`[Var_Default]` 优先**（整体预设可覆盖内联 `$default`）。

### B-3 Default 的生命周期

与 Schema 相同：创建新聊天时自动加载，存入 chat 层，与世界书解耦。

## 四、Initial 初始化

### C-1 语法

在消息内容中使用 `<Var_Initial>...</Var_Initial>` 包裹一段 JSON、YAML 或 TOML 格式的变量数据：

```xml
<Var_Initial>
角色:
  HP: 100
  MP: 50
  状态: "正常"
背包:
  - 名称: "铁剑"
    数量: 1
</Var_Initial>
```

### C-2 行为规则

- **清空当前层变量**：遇到 `<Var_Initial>` 时，当前楼层的变量不从上一层继承，而是被完全清空后用 Initial 内容赋值。
- **Default 补全**：Initial 赋值后，脚本检查 Schema 中声明但 Initial 中未提供的字段，用 Default 值填充。
- **Schema 校验**：补全完成后用 Schema 校验整体变量状态。
- **典型使用场景**：
  - 开场白中设定初始变量。
  - 对话中途的场景重置（如角色死亡重开、进入新地图等），此时 Initial 会清空旧变量、建立新状态。
- **回滚安全**：若用户对 Initial 产生的变量不满意，可直接删除该楼层，变量自动回退到上一层的状态。

### C-3 与 Update 的关系

- **可共存，按序执行**：同一条消息若同时含 `<Var_Initial>` 与 `<Var_Update>`，先 Initial（建立基线），再在该结果上执行 Update（增量修改）。

## 五、Update 变量更新

### D-1 语法

在消息内容中使用 `<Var_Update>...</Var_Update>` 包裹更新指令：

```xml
<Var_Update>
[
  { "op": "replace", "path": "/角色/HP", "value": 80 },
  { "op": "insert", "path": "/背包/-", "value": {"名称": "回复药水", "数量": 2} },
  { "op": "delete", "path": "/背包/0" }
]
</Var_Update>
```

> **标签的显示处理**：`<Var_Initial>` 和 `<Var_Update>` 标签及其内容会保留在聊天消息的原始文本中，脚本不负责隐藏或美化它们。若用户希望在聊天界面中隐藏这些标签，可通过 SillyTavern 的正则替换功能自行处理。

### 正文预处理

标签内正文在宽松 JSON 解析之前会经**文本清洗**，规则如下（均在对 BOM 处理并 trim 整段之后执行）：

- **多行 Markdown 围栏**：首行整行仅为 `` ``` `` 或 `` ```lang ``（该行不含其它反引号），且末行整行仅为 `` ``` `` → 去掉该首尾两行，视为剥除一层；同一轮清洗中至多执行 5 次。
- **单行紧凑形态**：整段以 `` ``` `` 或 `` ```lang `` 起首、以末尾 `` ``` `` 收尾，中间为数组或对象 → 去掉起首与末尾这一段围栏标记，视为剥除一层。
- **指令 JSON 截取**：取第一个 `[` 与最后一个 `]` 之间的子串（若无，则取第一个 `{` 与最后一个 `}`），作为下一步解析对象；用于去掉围栏外或数组/对象外的前后缀文字。围栏剥除仅依据上述行级与头尾形态，先于 JSON 解析执行。

### D-2 指令格式

| 操作 | 含义 | 路径不存在时 | 路径已存在时 |
|------|------|-------------|-------------|
| `replace` | 替换已有字段的值 | ❌ 报错 | ✅ 正常替换 |
| `insert` | 新增字段或向数组追加元素（`/-`） | ✅ 正常新增 | ❌ 报错（已存在） |
| `delete` | 删除字段或数组元素 | ❌ 报错 | ✅ 正常删除 |

路径格式使用 `/` 作为层级分隔符，与插值宏、`refer()` 共用同一套路径规范。

### D-3 多段合并

若同一条消息中包含多段 `<Var_Update>...</Var_Update>`，脚本将所有段中的指令数组合并为一个，按出现顺序执行。若合并后发现同一路径出现了多条指令，**仅最后一条生效**，之前的同路径指令被丢弃，并弹出**警告类** toastr（在面板为默认 `notice` 等级时仍会显示）。

### D-4 行为规则

- **继承机制**：执行 Update 前，若当前楼层尚无变量（即该消息未包含 `<Var_Initial>`），脚本从上一层消息深拷贝完整的变量状态到当前层，然后在拷贝上执行更新指令。
- **逐条执行与校验**：指令按数组顺序逐条执行，每条执行后用 Schema 校验当前变量状态。若某条指令导致校验失败，该条指令被丢弃（变量回滚到该条执行前的状态），继续执行后续指令。
- **变更记录**：每条成功执行的指令会生成一条变更记录（格式：`"旧值 → 新值"`），写入当前楼层的 `log` 字段。

**无 `<Var_Update>` 时的 `log`**：若某条消息最终仅通过「继承上一层 `data`」建立变量（无变量标签、或标签截断后略过解析等），该层**不会**执行任何 Patch，故须将本层 `log` 置为空对象。`log` 仅描述**本层**对 `<Var_Update>` 的解析结果，不从上一层复制，也不保留本层在先前一次解析中写入的旧 `log`（避免与「本轮未发生 Update」的语义冲突）。

## 六、数据存储模型

### E-1 存储架构

变量数据利用酒馆助手提供的多层作用域变量系统进行存储。酒馆助手的变量系统支持多个作用域层级（global、character、preset、script、chat、message），本脚本使用其中三个：

| 作用域 | 存储内容 | 生命周期 |
|--------|----------|----------|
| **global 层** | 脚本全局设置 | 跨角色卡、跨聊天永久持久化 |
| **chat 层** | Schema 声明（纯 JSON）、Default 值（纯 JSON） | 随当前聊天会话 |
| **message 层** | `data`（完整变量值）、`log`（本次变更记录） | 随单条消息 |

### E-2 message 层数据结构

每条消息的变量数据包含两个字段：

```json
{
  "data": {
    "角色": { "HP": 80, "MP": 50, "状态": "正常" },
    "背包": [{ "名称": "铁剑", "数量": 1 }]
  },
  "log": {
    "角色/HP": "100 → 80",
    "背包/-": "+ {名称: 回复药水, 数量: 2}"
  }
}
```

- **`data`**：变量的完整当前状态（真实值）。
- **`log`**：仅包含本次操作中发生变化的路径及其变更描述。未发生变化的路径不出现在此字段中。

酒馆助手的 message 层变量按 **swipe（消息分支）** 独立存储。同一条消息的不同 swipe 各自拥有独立的 `data` 和 `log`。用户在 SillyTavern 中切换 swipe 时，变量视图自动切换到对应 swipe 的状态。

### E-3 变量回滚

- 用户删除最新一条消息后，变量自动回退到上一条消息的 `data`，无需额外操作。
- 每层消息独立存储完整的变量快照，天然支持回滚。

## 七、校验与错误处理

### F-1 校验时机

| 时机 | 行为 |
|------|------|
| Initial 填充并补全 Default 后 | 校验整体变量，不合法的字段报错 |
| Update 每条指令执行后 | 校验整体变量，不通过则丢弃该条指令 |
| 用户在变量管理器中手动编辑 | 由酒馆助手变量管理器提供 UI 校验提示（非阻断式） |

### F-2 校验失败的处理

- 不合法的单条指令被丢弃，不影响该消息中后续指令的执行。
- 脚本统计本次解析中被丢弃的指令数（含校验失败、路径错误、操作冲突），根据容错阈值（见 J-1）判定严重程度：
  - **丢弃数 ≤ 容错阈值**：视为 **警告**（notice）。广播 `变量更新完成` 事件，按通知等级输出警告日志，用户可自行微调。
  - **丢弃数 > 容错阈值**：视为 **失败**（error）。广播 `变量更新失败` 事件，触发 Agents 脚本重试。
- **指令解析失败**（整段 `<Var_Update>` 经宽松解析仍无法得到有效指令数组）：直接视为 error，广播 `varupdate:update_failed`。
- **Initial 解析失败 / Schema 编译失败**：视为 error，但不广播重试事件（因为这不是 AI 的错，是角色卡配置问题）。仅输出错误日志，用户自行修正配置后点击「重新加载格式规则」。

## 八、消息监听与协作

### G-1 自动监听

VarUpdate 通过以下两个来源自动获知新消息的到达：

1. **SillyTavern 原生消息事件**（如 `MESSAGE_RECEIVED`）：覆盖酒馆原生生成流程产生的消息。使用数据到达事件而非渲染完成事件，确保第一时间获取消息内容。
2. **Agents 脚本的消息完成事件**：覆盖 Agents 流水线产生的消息（包括「仅解析」模式下不写入聊天记录的消息）。

收到消息后，脚本扫描其内容，识别是否存在**完整闭合**的变量标签：

- 存在完整闭合的 `<Var_Initial>...</Var_Initial>` 或 `<Var_Update>...</Var_Update>` → 执行解析和变量更新流程。
- 标签未闭合（如只有 `<Var_Update>` 而无 `</Var_Update>`）→ 视为无效，略过标签解析；若仍能确定目标楼层，则**仍执行继承**（见 D-4），避免本层变量悬空。
- 不包含任何变量标签 → **不解析标签**，但将本层变量设为**从前一层深拷贝 `data`**、`log` 置为空对象（与 D-4「继承机制」一致）；不广播 `varupdate:updated`。
- **标签名宽松匹配**：标准化时忽略大小写及下划线/连字符，例如 `<VarUpdate>`、`var_initial` 与标准写法等价。
- **代码块内忽略**：围栏 \`\`\`…\`\`\` 与行内 \`…\` 内的片段不作变量标签匹配，避免误识别。

脚本仅根据消息正文与上述规则维护变量链。

> **「仅解析」模式的变量归属**：Agents 脚本的「仅解析」步骤虽然不将生成结果写入聊天记录，但该步骤仍然是当前楼层生成流水线的一部分。因此，VarUpdate 从该步骤的输出中解析到的变量更新，写入当前最新楼层的 message 层变量中。

### G-2 事件通信

#### 广播事件（VarUpdate → 外部）

VarUpdate 向外广播以下事件，供其他脚本（如 Agents）按需消费（正式名以《接口与契约集中定义》为准）：

| 正式事件名 | 触发条件 | 负载要点 | Agents 预期响应 |
|------------|----------|----------|------------------|
| `varupdate:updated` | 本条存在 `<Var_Update>` 且应用成功，且丢弃数 ≤ 容错阈值 | `messageIndex`、`appliedCount`、`discardedCount`、`log` | ✅ 继续下一步 |
| `varupdate:update_failed` | Update 整段解析失败，或丢弃数 > 容错阈值等 | `messageIndex`、`reason`、`discardedCount` | ❌ 触发重试 |
| `varupdate:initialized` | Initial 校验通过并已写入 | `messageIndex`、`data` | 按需消费 |
| `varupdate:schema_ready` | Schema 编译成功 | `defNames` | 按需消费 |

#### 监听事件（外部 → VarUpdate）

VarUpdate 监听来自 Agents 等的协作事件，其中与「重试联动」直接相关的是：

| 正式事件名 | 来源 | VarUpdate 的响应 |
|------------|------|------------------|
| `varupdate:retry_requested` | Agents（或约定协作方） | 按负载回退指定楼层起的 message 层变量，避免与即将重算的正文不一致 |

该事件的含义是：上游判定当前步骤的输出不合格，即将重新生成。VarUpdate 收到后清理相应楼层的已写入变量。事件负载与常量名以《接口与契约集中定义》为准；消费方若需当前变量树应通过 `getVariables` 等 API 读取。

### G-3 典型协作场景

- Agents 流水线中的智能体 A 生成剧情文本，写入聊天记录。
- 智能体 B 分析剧情变化，输出含 `<Var_Update>` 的消息，写入聊天记录。
- VarUpdate 自动检测到新消息含变量标签，执行解析和更新。
- VarUpdate 广播 `varupdate:updated` 或 `varupdate:update_failed`。
- Agents 监听到失败后广播 `varupdate:retry_requested`，VarUpdate 回退变量。
- Agents 触发智能体 B 重新生成。

### G-4 边界划分

| 职责 | 归属 |
|------|------|
| 决定何时调用 AI、调用哪个模型、发送什么提示词 | **Agents 脚本** |
| AI 输出的接收和写入方式（追加/覆写/新楼层/仅解析） | **Agents 脚本** |
| 控制世界书条目在不同步骤中的注入（标签过滤） | **Agents 脚本** |
| 监听消息流，识别并解析 `<Var_Initial>` / `<Var_Update>` 标签 | **VarUpdate 脚本** |
| Schema 编译、变量校验、错误处理 | **VarUpdate 脚本** |
| 变量的持久化存储（读写酒馆助手变量系统） | **VarUpdate 脚本** |
| 变量回滚、检查点管理 | **VarUpdate 脚本** |

## 十、变量生命周期

### I-1 机制

随着对话增长，每层消息中存储的完整变量快照（`data`）会累积大量重复数据，导致聊天文件体积增大。脚本通过自动清理旧楼层的变量数据来控制文件大小。

### I-2 策略

- **保留层数**：保留最近 N 层消息的完整变量数据（N 在面板中配置，默认 20）。超过 N 层的旧楼层自动删除 `data` 和 `log`。
- **检查点豁免**：被标记为检查点的楼层始终保留，不受清理影响。
- **非破坏性**：清理操作仅删除变量数据，不影响消息内容本身。
- **自动恢复**：当用户删除消息导致最近楼层缺失变量时，脚本自动从最近的检查点或最近一条含有效变量的楼层开始逐层重新解析来恢复状态。

## 十一、脚本设置

### J-1 设置项

| 设置 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| 通知等级 | 下拉选择 | notice | 控制 toastr 弹窗和控制台输出的信息级别（见 J-2） |
| 自动初始化 | 开关 | 开启 | 开启时，自动识别开场白中的 `<Var_Initial>` 标签并执行变量初始化；关闭时需用户手动点击面板按钮触发初始化 |
| 容错阈值 | 数值 | 2 | 单次解析中被丢弃的指令数 ≤ 此值时视为警告，超过则视为失败并触发重试（见 F-2） |
| 变量生命周期 | 数值 | 20 | 保留最近 N 层的变量数据，超出部分自动清理（见第十章） |

### J-2 通知等级

| 等级 | toastr 弹窗 | 受等级过滤的控制台 |
|------|-------------|-------------------|
| `debug` | 调试（info）+ 成功 + 警告 + 错误 | 上述全部 + **trace**（见下） |
| `always` | 成功 + 警告 + 错误 | 成功 + 警告 + 错误 |
| `notice` | 警告 + 错误 | 警告 + 错误 |
| `error` | 仅错误 | 仅错误 |
| `silence` | 不显示常规通知 | 不输出常规日志 |

- **debug**：开发和排错时使用。`debug()` 会弹蓝色系 **info** toastr 并写控制台；**高频细节**（路径解析、路径修正、缓存命中、宏替换失败等）走 **trace**：**仅控制台、不弹 toastr**，避免刷屏。
- **always**：日常使用推荐。每次变量更新成功时弹出绿色通知，失败时弹出红色通知。
- **notice**（默认）：安静模式。仅在出现警告或错误时弹 toastr。
- **error**：最少打扰。仅在出错时通知。
- **silence**：常规 `debug` / `success` / `warning` / `error` 与 **trace**、变量快照摘要均不输出；**不弹**面板 `feedback`。**例外（便于排障）**：事件总线桥接失败、脚本顶层 `init` 失败仍会 **`console.error`**（带 `[VarUpdate][evt]` / `[VarUpdate][sys]`）；`silence` 下二者也**不弹** toastr。
- **feedback（面板手动操作）**：除 `silence` 外**不受**上表等级过滤，成功/失败都会 toastr + 控制台（默认归类 `man`）。

### J-3 控制台输出格式

常规日志在统一前缀 **`[VarUpdate]`** 上增加**场景码**，形如 **`[VarUpdate][场景码]`**，便于在控制台过滤。常用场景码：`boot` 启停、`msg` 消息与标签、`sch` Schema、`pat` 补丁管道、`wb` 世界书、`ui` 面板、`mac` 宏、`man` 手动操作、`life` 生命周期、`state` 变量快照、`evt` 事件桥接、`sys` 系统级初始化失败。

按类型着色（`%c`）：

| 颜色 | 含义 | 示例（场景码仅示意） |
|------|------|---------------------|
| 🔵 蓝色 | `debug()` 调试行（仅 **debug** 等级） | `[VarUpdate][sch] Schema: 编译成功…` |
| 灰蓝 | **trace** 高频细节（仅 **debug** 等级，无 toastr） | `[VarUpdate][pat] 路径修正: /旧 → /新` |
| 🟢 绿色 | 成功 | `[VarUpdate][pat] 变量更新完成: …` |
| 🟠 橙色 | 警告 | `[VarUpdate][pat] 同路径指令去重: …` |
| 🔴 红色 | 错误 | `[VarUpdate][pat] 指令解析失败: …` |

**变量快照（调试区）**：`debug` 等级打印完整对象；**always** 等级仅一行摘要（顶层键数）；**notice** 及以上不打印。

**toastr 标题**：与控制台一致，为 `[VarUpdate][场景码] 标题`（有场景码时）。

### J-4 设置存储

脚本设置存储在酒馆助手变量系统的 **global 层**，跨角色卡、跨聊天持久化。

## 十二、插值宏

### K-1 概述

VarUpdate 通过酒馆助手的 `registerMacroLike` 接口注册自定义插值宏，使角色卡作者可以在世界书条目和提示词中直接插入变量值。宏在消息发送前被替换为实际值，注入到发给 AI 的上下文中。

### K-2 宏格式

统一使用 `{{作用域/字段/变量路径}}` 格式，路径分隔符为 `/`，与 Update 指令、`refer()` 共用同一套路径规范：

```
{{message/data/角色/HP}}          → 80
{{message/data/角色/状态}}        → 正常
{{message/data/角色}}             → PromptalYAML 格式的角色对象
{{message/log/角色/HP}}           → 100 → 80
{{chat/schema}}                   → chat 层存储的 Schema 原始声明（JSON 对象）
{{chat/schema/$defs/物品}}        → 取 Schema 内子路径（示例）
{{chat/data/角色/HP}}             → chat 层 [Var_Default] 合并结果（键名为 default）下的路径
```

说明：`chat` 作用域下 **`data` 字段对应存储键 `default`**（世界书 `[Var_Default]`）；**Schema 在存储键 `schema` 上**，故读 Schema 应使用 `{{chat/schema/...}}`，勿写成 `{{chat/data/schema}}`（后者会在 Default 对象上误找 `schema` 键）。

**`message/data` 与 `$hide`**：仅 **`{{message/data/…}}`** 在序列化前根据**当前已编译**的 Schema 处理：`$hide` 节点在**概括**父对象或根 `data` 时被省略；若宏路径**正好**指向该 `$hide` 节点本身，则插入 **`\n`**（空白行）；若路径**更深一层**指向其子字段且该子字段在 Schema 上未标 `$hide`，则照常输出。`message/log`、`chat/*`、`global/*` 不应用此规则。无可用 Schema 时不裁剪。

### K-3 输出格式：PromptalYAML

宏输出复合类型变量值时，使用一种名为 **PromptalYAML** 的自定义结构化文本格式。该格式以 YAML 的缩进键值对语法为基础，针对 AI 上下文注入场景做了针对性改造，在保持结构清晰的同时最大程度节省 Token。

#### 基础类型输出

number / string / boolean 类型的变量直接输出原始值，不附加引号或格式标记。

#### 复合类型输出

object / array / record 类型的变量以 PromptalYAML 格式输出。核心规则：

- **键值对与嵌套**：使用 `键名: 值` 语法，嵌套层级通过两空格缩进表示。键名和字符串值均不加引号。
- **数组**：元素为基础类型时使用内联格式 `[元素1, 元素2, 元素3]`，元素为复合类型时使用展开格式（每个元素以 `- ` 前缀起始并缩进）。
- **空集合**：空数组输出为 `[]`，空对象输出为 `{}`。
- **多行字符串**：默认用三个反引号（`` ``` ``）包成一段围栏式块，块内字符按原样输出。若该字符串在去前导空白后**以** `` ``` `` **起首**，则宏序列化时**只**按 K-4 为各行补行首缩进，**不**再包一层 `` ``` ``。

#### 与标准 YAML 的差异

| 特性 | 标准 YAML | PromptalYAML |
|------|-----------|-------------|
| 多行字符串 | `\|` 或 `>` 标记，依赖缩进变化标识结束 | 默认 `` ``` `` 块；已以 `` ``` `` 起首的多行串仅缩进、不重复加外层块 |
| 字符串值 | 含特殊字符时需加引号 | 一律不加引号 |
| 基础类型数组 | 每个元素独占一行（`- ` 前缀） | 使用内联格式 `[a, b, c]` 节省行数 |
| 空集合 | 多种表示方式 | 统一使用 `[]` 和 `{}` |

#### 综合示例

````
角色:
  HP: 80
  MP: 50
  背景故事:
    ```
    曾经是王国的骑士团长，
    在一次战役中失去了右臂，
    从此隐居山林。
    ```
  技能列表: [火球术, 冰冻术, 雷击]
  装备:
    - 名称: 铁剑
      品质: 精良
    - 名称: 皮甲
      品质: 普通
  状态: 正常
背包容量: 20
````

### K-4 缩进对齐

宏替换时自动检测宏所在行前方的空格数，将 PromptalYAML 输出的每一行对齐到相同的缩进位置。例如：

```
当前角色状态：
  {{message/data/角色}}
```

替换后：

```
当前角色状态：
  HP: 80
  MP: 50
  状态: 正常
```

## 十三、变量条件标签

### L-1 概述

变量条件标签允许角色卡作者在预设条目、世界书条目和正则脚本的标题/备注中写入基于变量状态的条件表达式。VarUpdate 脚本在生成请求前读取当前变量状态，对条件求值，决定该条目是否参与本次提示词拼装。

- **定界符**：`[]`（半角方括号）。
- **标签位置不限**：标签可以放在文本的任何位置——开头、中间、末尾均可生效。脚本通过全文匹配检测标签是否存在，不要求固定位置。
- 一个条目可以携带多个标签（多个 `[]` 并排），脚本依次解析每一个。
- **变量来源**：标签条件中引用的变量读取自最新一层消息的 `data` 字段（即本脚本存储在 message 层的变量快照），与插值宏 `{{message/data/...}}` 读取同一个数据源。

> **与 Agents 智能体标签的关系**：姊妹脚本 Agents 使用 `【】`（全角方括号）管辖智能体匹配标签，与本脚本的变量条件标签互不干扰，各自独立解析。同一个条目可同时携带两种标签（如 `【创意写作】["HP" >= 60] 战斗设定`），分别由各自的脚本处理。两层过滤的结果取 AND——两者都通过才注入。

### L-2 逻辑组合规则

| 逻辑 | 语法 | 说明 |
|---|---|---|
| **OR（或）** | 同一个 `[]` 内用 `\|` 分隔 | 多项满足任一即可。例：`["地点" == "酒馆"\|"地点" == "旅店"]` |
| **AND（与）** | 多个 `[]` 并排 | 全部满足才注入。例：`["HP" >= 60]["MP" > 0]` |
| **NOT（非）** | `[]` 内首位加 `!` | 对整个条件取反。例：`[!"地点" == "虚空"]` |

多个变量条件标签并排时为 AND 关系——所有条件都满足时才通过。OR 仅在同一个 `[]` 内生效。

综合示例：

```text
["HP" >= 60]["MP" > 0][!"地点" == "虚空"|!"地点" == "深渊"]
```

含义：HP ≥ 60（AND）MP > 0（AND）地点既不是"虚空"也不是"深渊"时，注入该条目。

### L-3 基本比较

对变量值与字面量进行比较。语法为 `["变量名" 运算符 值]`。

| 运算符 | 含义 | 示例 |
|---|---|---|
| `==` | 宽松相等（不校验类型） | `["地点" == "酒馆"]` |
| `!=` | 宽松不等 | `["地点" != "酒馆"]` |
| `===` | 严格相等（校验类型） | `["HP" === 100]` |
| `!==` | 严格不等 | `["HP" !== "100"]`（排除字符串 "100"） |
| `>` | 大于 | `["好感度" > 0]` |
| `>=` | 大于等于 | `["好感度" >= 60]` |
| `<` | 小于 | `["好感度" < 80]` |
| `<=` | 小于等于 | `["好感度" <= 100]` |

数值比较（`>` `>=` `<` `<=`）适用于 number 类型的变量。字符串相等/不等（`==` `!=`）适用于任何类型。宽松比较（`==` `!=`）在类型不一致时尝试自动转换，严格比较（`===` `!==`）要求类型和值都匹配。

组合示例：`["好感度" >= 60]["好感度" < 80]至交好友` —— 好感度在 60（含）到 80（不含）之间时注入。

### L-4 通配符匹配

通配符不仅可用于右侧的值比对，也可直接用于左侧的变量路径寻址。

#### 1. 个数与位置规则

在任何涉及字符串匹配的场景中，都可以使用 `*` 遵循以下硬性规则：

- **1~2 个 `*`**：每个 `*` 严格匹配恰好 1 个字符。个数和位置都必须精确对应。
- **3 个及以上 `*`**：匹配任意数量的字符（0 到无穷）。

**位置规则：** 通配符在哪一侧，该侧就允许有额外内容；没有通配符的一侧必须是字符串的边界。

| 模式 | 含义 | 匹配 | 不匹配 |
|---|---|---|---|
| `"***开心***"` | 任意位置包含"开心" | "很开心啊"、"开心"、"我很开心" | — |
| `"***开心"` | 以"开心"结尾 | "很开心"、"开心" | "开心了" |
| `"开心***"` | 以"开心"开头 | "开心果"、"开心" | "很开心" |
| `"*开心*"` | 前后各恰好 1 字符 | "很开心啊" | "开心"、"超开心啊呀" |
| `"**民主主义"` | 前面恰好 2 字符 | "社会民主主义" | "民主主义"、"新民主主义" |

#### 2. 右侧值匹配

常用于：字符串比较（`==`、`!=`、`===`、`!==`）、数组含值（`∋`、`!∋`）、对象含键（`⊇`、`!⊇`）的值对比。
示例：`["当前地点" == "***森林***"]` 匹配“原始森林外围”。

#### 3. 左侧路径群查 (Implicit SOME)

您可以在左侧路径中嵌入通配符，让引擎按相同规则去扫描树的匹配键名。当且仅当找出多个匹配项时，触发 **隐式 Some 规则**：只要有一个分支满足后面的判定条件，整条语句就立判为真（True）。

> **注意：不要滥用单星号！**
> 因为 `*` 严格意味着“匹配恰好 1 个字符的长短”，所以 `["角色列表/*/生命值" > 0]` 只能找到名字正好等于 1 个字的那些角色（如“甲”）。
> 想要真正表达“所有子节点”的概念，必须写明是长字符串，即 `["角色列表/***/生命值" > 0]`（3颗星）。

- **值匹配示例**：`["队伍/***/状态标记" ⊇ "中毒"] 全员解药激活`
  - 解释：引擎遍历“队伍”下所有（不限键名长度的）节点。如果*任何一个*成员带有“中毒”标记，条件为真。
- **长度与存在性衍生**：
  - `["背包/***宝石" # > 0]`：探测背包里长短任意但以“宝石”结尾的物品集合中，是否有*任意一个*的具体数量（#获取其长度）大于 0。
  - `["记忆系统/旧事***" !?]`：因为 `!?` 代表不存在，在群查模式下，它表示“连一根毛都没捞到”。即：没有任何一项属性叫“旧事起头”的节点存在时（返回集合为空），条件为真。
通配符规则与本脚本 Schema 约束中的 `$enum` 通配符完全一致（见二·A-3）。

组合示例：`["心情" == "***开心***"|"心情" == "***高兴***"]开心时表现` —— 心情变量中包含"开心"或"高兴"时注入。

### L-5 存在性检查

检查变量是否已被定义（存在且非 null / undefined）。

| 语法 | 含义 | 示例 |
|---|---|---|
| `["变量名" ?]` | 变量已定义时为真 | `["圣杯" ?]` |
| `["变量名" !?]` | 变量未定义时为真 | `["圣杯" !?]` |

用例：`["圣杯" ?]圣杯持有者的特殊剧情` —— 只有在变量"圣杯"被创建之后才注入这段设定。

### L-6 嵌套属性访问

变量为对象类型时，使用 `/` 作为路径分隔符访问深层属性。适用于所有运算符。

```text
["角色/状态/HP" >= 60]
["角色/装备/武器" == "圣剑"]
["世界/天气" == "暴风雨"]
["角色/技能列表" ∋ "***火焰***"]
```

路径中的每一段对应对象的一个层级。若路径中任何一层不存在，条件判定为假（不报错，静默不匹配）。

此处的 `/` 路径格式与本脚本在 Update 指令路径、`refer()` 引用和插值宏中使用的路径格式相同，共用同一套路径规范。

### L-7 数组操作

| 运算符 | 含义 | 语法 | 说明 |
|---|---|---|---|
| `∋` | 数组含值 | `["物品栏" ∋ "圣杯"]` | 按值查找：数组中是否存在等于指定值的元素 |
| `!∋` | 数组不含值 | `["物品栏" !∋ "圣杯"]` | 按值排除 |

值支持通配符：`["物品栏" ∋ "***圣杯"]` —— 物品栏中是否有以"圣杯"结尾的元素。

### L-8 对象/字典操作

| 运算符 | 含义 | 语法 | 说明 |
|---|---|---|---|
| `⊇` | 对象含键 | `["属性" ⊇ "力量"]` | 按键名查找：对象中是否存在指定键 |
| `!⊇` | 对象不含键 | `["属性" !⊇ "力量"]` | 按键名排除 |

键名支持通配符：`["物品栏" ⊇ "***元素魔法书"]` —— 物品栏对象中是否有键名以"元素魔法书"结尾的条目。

### L-9 集合长度判断

使用 `#` 运算符获取数组或对象的元素数量，再与数值进行比较。

| 语法 | 含义 |
|---|---|
| `["物品栏" # >= 3]` | 物品栏至少有 3 个元素 |
| `["队伍成员" # == 4]` | 队伍恰好 4 人 |
| `["物品栏" # > 0]` | 物品栏非空 |

`#` 后接任意数值比较运算符（`>` `>=` `<` `<=` `==` `!=`）。

### L-10 变量与变量比较

比较两个变量的值（而非变量与字面量）。使用 `$"变量名"` 语法标记右侧为变量引用。

| 语法 | 含义 |
|---|---|
| `["当前HP" <= $"最大HP"]` | 当前 HP 的值 ≤ 最大 HP 的值 |
| `["地点" == $"目标地点"]` | 当前地点等于目标地点 |
| `["攻击力" > $"敌人/防御力"]` | 我方攻击力 > 敌人的防御力（支持嵌套路径） |

右侧的 `$"..."` 同样支持 `/` 路径访问嵌套属性。

### L-11 作用范围

变量条件标签管辖以下三类条目的注入，标签写在各条目的标识字段中：

| 条目类型 | 标签写入字段 | 示例 |
|---|---|---|
| **世界书条目** | 备注（comment） | `["HP" >= 60] 战斗设定` |
| **预设条目** | 标题（name） | `["好感度" >= 80] 亲密系统提示词` |
| **正则脚本** | 名称（script_name） | `["地点" == "战场"] 战斗描写格式化` |

### L-12 判定流程

变量条件标签是在酒馆原生条目管理之上的**附加过滤层**。完整判定流程为：

```text
1. 酒馆原生判定（条目启用/关闭、关键词激活等）
   ↓ 通过
2. Agents 智能体标签判定（如 Agents 脚本已启用）
   ↓ 通过
3. VarUpdate 变量条件标签判定
   ↓ 通过
4. 最终注入
```

- 前置层未通过的条目不会到达变量条件判定层。
- 未携带任何变量条件标签的条目在本层自动通过（不过滤）。
- VarUpdate 独立运行（无 Agents）时，跳过步骤 2，直接由步骤 1 进入步骤 3。

### L-13 条件求值失败的处理

- **变量路径不存在**：条件判定为假（静默不匹配），不报错。
- **变量数据为空**（尚未初始化）：所有变量条件标签判定为假，条目不注入。
- **表达式解析失败**（语法错误）：该条标签判定为假，并输出警告日志（受通知等级控制）。不影响同条目上其他标签的解析。

### L-14 完整运算符速查表

| 类别 | 运算符 | 语法示例 | 说明 |
|---|---|---|---|
| 宽松相等 | `==` | `["地点"=="酒馆"]` | 不校验类型 |
| 宽松不等 | `!=` | `["地点"!="酒馆"]` | 不校验类型 |
| 严格相等 | `===` | `["HP"===100]` | 校验类型 |
| 严格不等 | `!==` | `["HP"!=="100"]` | 校验类型 |
| 大于 | `>` | `["HP">0]` | number 类型 |
| 大于等于 | `>=` | `["HP">=60]` | number 类型 |
| 小于 | `<` | `["HP"<80]` | number 类型 |
| 小于等于 | `<=` | `["HP"<=100]` | number 类型 |
| 存在性 | `?` / `!?` | `["圣杯"?]` | 变量是否已定义 |
| 通配符 | `*` | `["x"=="***火***"]` | 所有字符串场景可用 |
| 数组含值 | `∋` / `!∋` | `["背包"∋"剑"]` | 按值查找 |
| 对象含键 | `⊇` / `!⊇` | `["属性"⊇"力量"]` | 按键名查找 |
| 集合长度 | `#` + 比较 | `["背包"#>=3]` | 元素数量判断 |
| 嵌套访问 | `/` 路径 | `["角色/HP">=60]` | 深层属性 |
| 变量引用 | `$"var"` | `["HP"<=$"MaxHP"]` | 变量与变量比较 |
| OR | `\|` | `["x"==1\|"y"==2]` | 同一 `[]` 内满足任一 |
| AND | 多个 `[]` | `["x">=1]["y">=1]` | 全部满足才注入 |
| NOT | `!` 前缀 | `[!"HP">=60]` | 对条件取反 |
