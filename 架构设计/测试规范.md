# VarUpdate 测试规范

本文档约束 **本仓库** 内测试的分层、各测试文件应覆盖的行为范围、集成与手动验收及覆盖率期望。编写用例时对照《基础设施功能卡》模块边界与《面向用户功能卡》行为；事件名与负载形状以《接口与契约集中定义》为准。

---

## 一、分层

| 层级 | 位置 / 方式 | 目的 |
|------|----------------|------|
| 单元 | `tests/**/*.test.ts`（不含 `integration/`） | 纯逻辑、Node 可运行、输入输出可断言 |
| 集成 | `tests/integration/*.test.ts` | Stub 全局 API 后串联多模块或主流程 |
| 手动 | SillyTavern + 酒馆助手 | DOM、世界书、宏、真实 `getVariables` / 事件时序 |

## 三、通用约定

1. 测试与 `src/` 平级放在 `tests/`；文件名为 `*.test.ts`。
2. `describe` / `it` 标题优先中文，与功能卡章节号可交叉引用（如「五·D-3 同路径去重」）。
3. 每个模块：**边界与失败路径优先**，再补主路径。
4. 不得在测试中依赖真实 iframe；宿主 API 用 `vi.stubGlobal` 或集成文件内 Map 替身。

## 四、分文件覆盖要求

以下树状列为 **应覆盖要点**：与当前 `it` 名称不必一一对应，但缺失时应在补测或 CR 中说明理由。

### 4.1 `format-parser.test.ts` → `src/modules/format-parser.ts`

**两个入口（实现不同，须分别写清）：**

- **`parseStructuredText`（async）**：顺序 **JSON → TOML → YAML**；空串 → `FormatParseError`；三者均失败 → `FormatParseError`，且 `details` 含 `jsonError` / `tomlError` / `yamlError`（用于 `formatParseErrorDetails`）。
- **`parseStructuredTextSync`**：顺序 **JSON → YAML**（**不含 TOML**，与 async 不同）；空串与双失败错误信息路径与 async 区分。

**共同行为：**

```
├── 合法 JSON 对象 → 原样对象
├── JSON 数组 / 标量 → { _value: … }（含 null → {} 等与 wrapIfNotObject 一致）
├── YAML 对象 / 嵌套键
├── TOML（仅 async 入口）：`[section]`、`key = value` 等可解析样例
├── 非对象顶层结果 → _value 包裹
└── formatParseErrorDetails：多行拼接含各解析器错误（有则输出）
```

**缺口提示**：若单测仅覆盖 `parseStructuredTextSync`，应至少增加若干 `parseStructuredText` 用例覆盖 **TOML 分支** 与 **三失败** 的 `details` 结构。

## 五、集成测试 `tests/integration/pipeline.test.ts`

**目标**：在 Node 内 stub `getVariables` / `replaceVariables`、必要时 `eventEmit` / `eventOn`、`SillyTavern.getContext` 等，跑通「提取标签 → 解析 → 编译或校验 → Patch → 写回」的缩略真链。

**占位用例（与文件中 `it.todo` 对齐，实现后改为断言）：**

```
├── Schema → Initial → Update 完整流程
├── 错误传播
│   ├── FormatParseError → 不写变量
│   ├── Patch 部分失败 → discarded 计数与阈值语义
│   └── Schema 单条失败 → 回滚该条
├── 跨模块
│   ├── refer() 读当前 data
│   ├── 宏侧读存储（若在此层测）
│   └── $hide 宏输出
├── 变量条件标签
│   ├── 条件标签过滤世界书条目（模块 10 + 11 联动）
│   ├── 变量更新后过滤结果变化
│   └── 无 Schema 时标签求值仍正常工作
└── 生命周期 API：clearMessageVariablesAfter、pruneOrphanMessageVariables
```

**Mock 清单（按场景逐步加）：** `getVariables`、`replaceVariables`、`_`（cloneDeep）、`eventOn`/`eventEmit`、`registerMacroLike`、`parent.toastr`、`z`（safeParse）、精简 `chat[]`。

## 六、手动验收清单（宿主内）

- [ ] 新聊天 / 切换聊天：世界书 `[Var_Schema]` / `[Var_Default]` 加载、编译成功或错误提示与 chat 层 `schemaStatus` 一致。
- [ ] 开场白 `autoInitialize` 开/关：是否自动跑第 0 层标签逻辑。
- [ ] `<Var_Initial>`：Default 补全、Schema 校验、message 层 `data` / `log` / `isInitPoint`。
- [ ] `<Var_Update>`：继承上一层、逐条校验与回滚、同路径去重 toastr、`log` 格式。
- [ ] 容错阈值：丢弃数低于/高于阈值时写入与 `varupdate:updated` / `update_failed`（若接 Agents）。
- [ ] 插值宏：`message` / `chat` / `global` 路径、`chat` 下 schema 与 default 键名、$hide 行为。
- [ ] 面板五键 + 魔棒两键：重载规则、开场白重初始化、当前层重解析、检查点、链式重解析。
- [ ] 通知等级与 `feedback`；`silence` 下 bridge/bootstrap 仍可在控制台看到错误。
- [ ] 删消息、swipe、MESSAGE_EDITED：变量与调试区是否一致。
- [ ] `retentionDepth` 与检查点：远期清理与豁免。
- [ ] 变量条件标签：世界书条目 comment 中含 `["HP" >= 60]` 时，HP 低于 60 该条目不注入。
- [ ] 变量条件标签：预设条目 name 字段中的条件标签正常过滤。
- [ ] 变量条件标签：正则脚本 scriptName 中的条件标签正常过滤。
- [ ] 变量条件标签：无标签条目不受影响（自动通过）。
- [ ] 变量条件标签：变量数据未初始化时，带标签条目全部不注入。
