# 测试规范——MyProject 统一测试策略

> 本文档定义 Agents 和 VarUpdate 两个项目及五个公共库的统一测试策略、工具链和测试用例规范。
> 本文档是两个项目共享的，各持一份副本（若修改需同步更新），与 `代码规范与工程细节.md` 同级。

---

## 一、总体策略

### 测试分层

```text
┌────────────────────────────────────────┐
│  第三层：端到端联调（手动）              │  两个脚本在酒馆环境中协同运作
├────────────────────────────────────────┤
│  第二层：模块集成测试                   │  模块间的数据流和事件通信
├────────────────────────────────────────┤
│  第一层：单元测试（自动化）             │  纯函数 / 纯逻辑模块
└────────────────────────────────────────┘
```

### 测试优先级

自动化测试集中在**第一层**——对纯函数和纯逻辑模块编写单元测试。这些模块不依赖酒馆环境，输入输出可控，且包含最多的边界情况。

第二层集成测试通过 mock 酒馆 API 进行，覆盖关键数据流。第三层端到端测试在真实酒馆环境中手动执行。

### 原则

1. **测试与源码同级**：测试文件放在 `tests/` 目录，与 `src/` 平级。
2. **按模块组织**：每个模块一个测试文件（如 `tests/format-parser.test.ts`）。
3. **命名规范**：测试描述使用中文（与文档一致），便于直接对照功能卡。
4. **边界用例优先**：每个测试模块先覆盖边界/异常情况，再补充正常路径。

## 二、工具链

| 工具 | 用途 | 选型理由 |
|------|------|---------|
| **Vitest** | 测试框架 + 断言库 | 原生 TypeScript + ESM 支持，与 esbuild 生态兼容 |
| **vitest --coverage** | 覆盖率统计 | 内置，无需额外配置 |

### 配置

```typescript
// vitest.config.ts（各项目根目录）
import { defineConfig } from 'vitest/config';

export default defineConfig({
  test: {
    include: ['tests/**/*.test.ts'],
    environment: 'node',
    globals: true,
  },
});
```

### 命令

```json
// package.json scripts
{
  "test": "vitest run",
  "test:watch": "vitest",
  "test:coverage": "vitest run --coverage"
}
```

## 三、公共库测试规范

> 公共库是纯函数库，无外部依赖，**必须 100% 单元测试覆盖**。

### 3.1 wildcard-match

```
tests/wildcard.test.ts
├── 基础匹配
│   ├── 精确匹配相同字符串 → true
│   ├── 不同字符串 → false
│   └── 空字符串 vs 空模式 → true
├── * 通配符
│   ├── 单独 "*" 匹配任意字符串 → true
│   ├── "abc*" 匹配 "abcdef" → true
│   ├── "*abc" 匹配 "xyzabc" → true
│   ├── "a*c" 匹配 "abc", "axyzc" → true
│   ├── "a*c" 不匹配 "abcd" → false
│   └── 多个 "*"："a*b*c" 匹配 "axbxc" → true
├── ? 通配符
│   ├── "a?c" 匹配 "abc" → true
│   ├── "a?c" 不匹配 "ac" 或 "abbc" → false
│   └── "???" 匹配恰好三个字符 → true, 其他长度 → false
├── 大小写敏感性
│   └── 默认大小写不敏感
└── 边界
    ├── 模式为空字符串 → 仅匹配空字符串
    ├── 文本含特殊字符（中文、emoji）→ 正常匹配
    └── 极长字符串 → 不超时
```

### 3.4 schema-to-zod

```
tests/schema-to-zod.test.ts
├── 基础类型映射
│   ├── $type: number → z.number()
│   ├── $type: integer → z.number().int()
│   ├── $type: string → z.string()
│   ├── $type: boolean → z.boolean()
│   ├── $type: any → z.any()
│   └── $type: object → z.object().strict()
├── force 变体
│   ├── number(force) + 字符串 "42" → 通过，转为 42
│   ├── integer(force) + 字符串 "3.7" → 失败（非整数）
│   └── string(force) + 数字 42 → 通过，转为 "42"
├── 复合类型
│   ├── array<number> → z.array(z.number())
│   ├── record<string> → z.record(z.string(), z.string())
│   └── 联合类型 [number, string] → z.union()
├── 约束
│   ├── $min / $max → 数值范围校验
│   ├── $enum → 枚举校验（含通配符）
│   ├── $optional → .optional()
│   ├── $default → .default(value)
│   ├── $minLength / $maxLength → 字符串长度
│   ├── $regex → 正则校验
│   ├── $minItems / $maxItems → 数组/record 长度
│   ├── $key_rule → record 键名约束
│   ├── $extensible: true → .passthrough()
│   ├── $either → z.union()
│   └── $!前缀 → .refine() 取反
├── $defs
│   ├── 定义结构体 + 引用 → 正确解析
│   ├── 引用不存在的结构体名 → SchemaCompileError
│   └── $defs 可在 array<> 和 record<> 中引用
├── refer()
│   ├── refer(路径) 作为 $max 值 → 延迟求值
│   ├── refer() 引用不存在的路径 → 静默跳过（不报错）
│   └── refer() 引用的值在运行时变化 → 每次校验都重新读取
└── 错误收集
    ├── 多个错误不中断 → 一次性收集所有错误
    └── 错误信息包含路径 + 类型 + 描述
```

> 底层解析能力已在公共库 flexible-json-patch 中测试。模块 3 聚焦于反向路径解析和执行校验。

```
tests/json-patch.test.ts

│   ├── 叶子节点唯一 → 直接匹配
│   ├── 叶子节点歧义 → 逐级消歧义

│   ├── 叶子节点不存在 + insert → 正常新增

## 八、覆盖率目标

| 类别 | 目标 | 说明 |
|------|:----:|------|
| 5 个公共库 | **≥ 95%** | 纯函数，无理由不覆盖 |
| VarUpdate 模块 1-6 | **≥ 80%** | 核心逻辑模块 |
| Agents 模块 1-5 | **≥ 80%** | 核心逻辑模块 |
| UI 模块（9） | 不要求 | DOM 操作，通过端到端手动验证 |
| 事件总线/通知（7/8） | **≥ 60%** | 薄封装层，关键路径覆盖即可 |
