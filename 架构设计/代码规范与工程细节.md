## 一、构建与工程化

### 1.1 打包工具：esbuild

两个主项目和五个辅助库均使用 **esbuild** 打包，输出 **ESM**（ES Module）格式。

#### esbuild 配置要点

```typescript
// 主项目构建入口：Agents 多为 esbuild.config.ts；本仓库 VarUpdate 为 esbuild.config.mjs（逻辑等价，见该文件内 vendor alias）
import { build } from 'esbuild';

build({
  entryPoints: ['src/index.ts'],
  bundle: true,
  outfile: 'dist/bundle.js',
  format: 'esm',               // ES Module 格式，支持顶层 import/export
  platform: 'browser',
  target: 'es2020',
  sourcemap: true,
  minify: false,                // 开发时不压缩，发布时改 true
  external: [                   // CDN 或全局提供的库，不打入 bundle
    'klona',
    'smol-toml',
    'json5',
    // 五个辅助库也标记为 external（通过 import map 或 CDN URL 解析）
  ],
});
```

```typescript
// esbuild.config.ts（辅助库：label-parser / wildcard-match / 等）
build({
  entryPoints: ['src/index.ts'],
  bundle: true,
  outfile: 'dist/index.js',
  format: 'esm',               // 辅助库同样输出 ESM
  platform: 'browser',
  target: 'es2020',
  minify: true,                 // 辅助库始终压缩
});
```

> **为什么选 ESM 而非 IIFE**：酒馆助手 iframe 环境支持 ES Module 的顶层 `import`。参考脚本已证实此模式可用：
> ```javascript
> // 参考：[七日末途] 注册变量类型脚本
> import { registerMvuSchema } from 'https://testingcf.jsdelivr.net/gh/StageDog/tavern_resource/dist/util/mvu_zod.js';
> ```

#### 为什么不用 webpack

参考项目 MVU 使用了 webpack（因为它用了 Vue + CSS loader），但我们的项目不使用 Vue，UI 用 jQuery + 原生 DOM 构建，不需要 loader 生态。esbuild 原生支持 TypeScript，打包速度快 10-100 倍，配置极简。

### 1.3 依赖引入策略

#### 四种引入方式

| 来源 | 引入方式 | 包含 |
|------|---------|------|
| **iframe 全局** | 直接使用，不需任何安装 | `_`(lodash)、`z`(Zod)、`YAML`(js-yaml)、`toastr`、`$`(jQuery)、`showdown` |
| **酒馆助手 API** | 全局函数/对象 | `TavernHelper`、`eventEmit/On/Once/ClearAll`、`SillyTavern` |
| **酒馆本体模块** | `TavernHelper.import()` 动态加载 | `extension_settings`、`promptManager`、世界书 API |
| **CDN import** | `import ... from 'https://...'` | 五个辅助库、`klona`、`smol-toml`、`json5` |

#### CDN import 详解

所有不由 iframe 全局提供的第三方库和自有辅助库，统一通过 **jsDelivr CDN** 运行时加载。

```typescript
// ─── 五个辅助库（自有，托管在 GitHub） ───
import { parseLabels } from 'https://testingcf.jsdelivr.net/gh/<用户名>/<仓库名>/label-parser/dist/index.js';
import { wildcardMatch } from 'https://testingcf.jsdelivr.net/gh/<用户名>/<仓库名>/wildcard-match/dist/index.js';
import { compileSchema } from 'https://testingcf.jsdelivr.net/gh/<用户名>/<仓库名>/schema-to-zod/dist/index.js';
import { applyPatch } from 'https://testingcf.jsdelivr.net/gh/<用户名>/<仓库名>/flexible-json-patch/dist/index.js';
import { stringify } from 'https://testingcf.jsdelivr.net/gh/<用户名>/<仓库名>/promptal-yaml/dist/index.js';

// ─── 第三方库（npm 包，通过 jsDelivr npm CDN） ───
import { klona } from 'https://testingcf.jsdelivr.net/npm/klona/+esm';
import TOML from 'https://testingcf.jsdelivr.net/npm/smol-toml/+esm';
import JSON5 from 'https://testingcf.jsdelivr.net/npm/json5/+esm';
```

> **优势**：
> - 更新辅助库只需推送 GitHub，无需重新打包主项目
> - AI 聊天本来就要联网，不增加额外的网络依赖
> - 主项目 bundle 体积极小（仅包含业务代码）
>
> **jsDelivr CDN URL 格式**：
> - GitHub 仓库：`https://testingcf.jsdelivr.net/gh/用户名/仓库名/文件路径`
> - npm 包（ESM）：`https://testingcf.jsdelivr.net/npm/包名/+esm`
> - 锁定版本：`/gh/用户名/仓库名@版本号/文件路径`

#### iframe 全局变量声明

```typescript
/**
 * 酒馆助手的 predefine.js 会将以下对象从宿主页面注入到 iframe window 中：
 * _          → lodash（完整库）
 * YAML       → js-yaml（已全局可用，无需导入）
 * z          → Zod（已全局可用，无需导入）
 * toastr     → 通知弹窗
 * showdown   → Markdown 转 HTML
 * $          → jQuery（通过 parent_jquery.js 注入）
 * TavernHelper → 酒馆助手核心 API
 * eventEmit / eventOn / eventOnce / eventClearAll → 事件总线（从 TavernHelper._bind 解构）
 * SillyTavern → 酒馆本体上下文（通过 defineProperty 代理）
 *
 * 注意：klona 不在此列表中。使用 CDN import 或用 _.cloneDeep 替代。
 */
declare const _: typeof import('lodash');
declare const YAML: typeof import('js-yaml');
declare const z: typeof import('zod');
declare const toastr: { success: Function; error: Function; warning: Function; info: Function };
declare const $: typeof import('jquery');

declare const TavernHelper: {
  import: (module: string) => Promise<any>;
  // ... 其他 API 见契约文档第二章
};

// 事件总线函数（从 TavernHelper._bind 解构并绑定到 iframe window）
declare function eventEmit(eventName: string, ...args: any[]): Promise<void>;
declare function eventOn(eventName: string, handler: Function): void;
declare function eventOnce(eventName: string, handler: Function): void;
declare function eventClearAll(): void;
```

## 二、错误处理模式

### 2.1 错误分级

所有错误按**后果严重性**分为三级：

| 等级 | 名称 | 后果 | 处理方式 | 示例 |
|------|------|------|---------|------|
| **L1** | 静默容错 | 不影响核心流程 | `notify('debug', ...)` + 继续执行 | 单条 Patch 指令校验失败（丢弃该条，继续后续） |
| **L2** | 通知用户 | 影响当前步骤但不致命 | `notify()` + 继续/重试 | API 调用超时（触发重试）、Schema 编译警告 |
| **L3** | 中止流程 | 无法继续的致命错误 | `feedback()` + 抛出异常中止 | 所有重试耗尽、配置数据损坏、API Key 无效 |

### 2.2 自定义错误类

两个项目共享相同的错误类层次结构：

```typescript
/**
 * 所有自定义错误的基类
 * 附带 context 字段用于结构化错误信息
 */
class ScriptError extends Error {
  constructor(
    message: string,
    public readonly context: Record<string, any> = {}
  ) {
    super(message);
    this.name = this.constructor.name;
  }
}

// ─── Agents 专用 ───
class ApiCallError extends ScriptError {}      // API 调用失败（网络/认证/格式）
class PipelineAbortError extends ScriptError {} // 流水线被中止（用户手动/重试耗尽）
class ConfigError extends ScriptError {}        // 配置数据无效

// ─── VarUpdate 专用 ───
class FormatParseError extends ScriptError {}   // YAML/JSON/TOML 均无法解析
class SchemaCompileError extends ScriptError {} // Schema 编译失败
class PatchExecuteError extends ScriptError {}  // Patch 指令执行失败
```

### 2.3 catch 策略

#### 规则 1：在边界处 catch，不在内部 catch

模块内部的函数**不应自行 catch 异常**，而是让异常向上冒泡到**调用方**。调用方根据上下文决定如何处理。

```typescript
// ❌ 错误做法：模块内部吞掉异常
function parseText(text: string) {
  try {
    return JSON.parse(text);
  } catch {
    return null;  // 吞掉了错误信息
  }
}

// ✅ 正确做法：模块内部抛出，调用方处理
function parseText(text: string): Record<string, any> {
  // 尝试 JSON → TOML → YAML，全部失败则抛出
  throw new FormatParseError('三种格式均无法解析', { jsonErr, tomlErr, yamlErr });
}

// 调用方（主控制器）
try {
  const data = parseText(rawText);
} catch (e) {
  if (e instanceof FormatParseError) {
    notify('error', '格式解析失败', e.message);
    return;  // 不中止整个流程，跳过这条消息
  }
  throw e;  // 未知异常继续向上冒泡
}
```

#### 规则 2：只 catch 你知道如何处理的异常

```typescript
try {
  await apiCall(config);
} catch (e) {
  if (e instanceof ApiCallError) {
    // 已知异常：按重试逻辑处理
    retryController.handleFailure(e);
  } else {
    // 未知异常：不吞，向上抛出
    throw e;
  }
}
```

#### 规则 3：async 函数中 catch 后不要忘记 return 或 throw

```typescript
// ❌ catch 后没有 return，函数继续执行后续逻辑
async function process() {
  try {
    await step1();
  } catch (e) {
    notify('error', '步骤1失败', e.message);
    // 缺少 return! step2 会继续执行
  }
  await step2();
}

// ✅ catch 后明确 return
async function process() {
  try {
    await step1();
  } catch (e) {
    notify('error', '步骤1失败', e.message);
    return;  // 明确中止
  }
  await step2();
}
```

### 2.4 错误传播路线

```text
模块内部（抛出 ScriptError 子类）
  ↓ 冒泡
模块公开函数（不 catch，让异常穿透）
  ↓ 冒泡
主控制器 / 流程编排层（catch 并决策）
  ├─ L1：warn + 继续
  ├─ L2：notify + 重试 / 跳过
  └─ L3：feedback + 中止流程
```

## 三、模块间通信模式

### 3.1 同脚本内部：直接 import

同一脚本内的模块之间使用 **ES Module import** 直接调用，不走事件总线。

```typescript
// ✅ 脚本内部模块间：直接 import
import { parseStructuredText } from './modules/format-parser';
import { notify } from './modules/notification';

const data = parseStructuredText(rawText);
notify('debug', '解析完成', `得到 ${Object.keys(data).length} 个字段`);
```

事件总线 **仅用于跨脚本通信**（Agents ↔ VarUpdate）和**酒馆本体事件监听**。脚本内部不使用事件解耦（过度解耦会增加调试难度、破坏类型安全）。

### 3.2 异步处理：统一 async/await

所有异步操作使用 **async/await**，不使用 `.then()` 链。

```typescript
// ✅ async/await
async function runStep(agent: AgentConfig): Promise<StepResult> {
  const prompt = await buildPrompt(agent);
  const response = await callApi(agent, prompt);
  const processed = postProcess(response, agent.postProcess);
  return processed;
}

// ❌ .then() 链
function runStep(agent: AgentConfig) {
  return buildPrompt(agent)
    .then(prompt => callApi(agent, prompt))
    .then(response => postProcess(response, agent.postProcess));
}
```

**例外**：事件监听回调中的异步操作可使用 void async IIFE：

```typescript
eventBus.on('agents:message_complete', (payload) => {
  void (async () => {
    await processMessage(payload);
  })();
});
```

### 3.3 状态管理：模块级单例

每个有状态的模块使用**模块级变量**维护状态（利用 ES Module 的单例特性），对外暴露 getter/setter 函数：

```typescript
// modules/notification.ts
let currentLevel: NotifyLevel = 'notice';

export function setLevel(level: NotifyLevel): void {
  currentLevel = level;
}

export function notify(level: NotifyLevel, title: string, msg: string): void {
  if (levelValue(level) < levelValue(currentLevel)) return;
  // ...
}
```

不使用 class 单例模式（class 增加了不必要的 `this` 绑定复杂度），不使用全局变量（避免跨模块耦合）。

### 3.4 初始化顺序

主控制器 `index.ts` 负责按依赖顺序初始化所有模块：

```typescript
// src/index.ts
export async function init() {
  // 1. 无依赖模块先初始化
  notificationModule.setLevel(globalSettings.notifyLevel);

  // 2. 注册事件监听
  eventBus.setupListeners();

  // 3. 加载持久化配置
  await configStore.load();

  // 4. UI 最后初始化（依赖配置数据）
  uiPanel.render();
}
```

## 四、命名约定

### 4.1 文件名

| 类型 | 规则 | 示例 |
|------|------|------|
| 模块文件 | `kebab-case.ts` | `label-engine.ts`、`format-parser.ts` |
| 类型定义 | `kebab-case.ts`（置于 `types/`） | `types/index.ts` |
| 数据文件 | `kebab-case.json` | `api-params-map.json` |
| 目录 | `kebab-case` | `schema-compiler/`、`json-patch/` |

### 4.2 代码命名

| 元素 | 规则 | 示例 |
|------|------|------|
| 变量 / 函数 | `camelCase` | `parseText`、`currentLevel` |
| 类 / 接口 / 类型 | `PascalCase` | `AgentConfig`、`SchemaCompileError` |
| 常量 | `UPPER_SNAKE_CASE` | `SCHEMA_KEYWORDS`、`OP_ALIASES` |
| 事件名 | `namespace:snake_case` | `agents:pipeline_started`、`varupdate:updated` |
| 存储键名 | `PascalCase_snake_case` 混合 | `Agents_config`、`VarUpdate_config` |

### 4.3 注释

- **公共函数/接口**：必须有 JSDoc 注释，包含 `@param` 和 `@returns`
- **模块顶部**：简述该模块的职责、上下游依赖
- **复杂逻辑**：行内注释解释「为什么」而非「做了什么」
- **TODO / FIXME**：格式 `// TODO(模块名): 描述`

### 4.4 导出规则

- 每个模块导出**具名函数**（`export function`），不导出 default
- 模块目录（如 `schema-compiler/`）通过 `index.ts` 统一导出
- 类型定义通过 `export type` / `export interface` 导出

```typescript
// ✅ modules/schema-compiler/index.ts
export { compileSchema } from './compiler';
export { validateWithSchema } from './validator';
export type { CompiledSchema, ValidationResult } from './types';
```
