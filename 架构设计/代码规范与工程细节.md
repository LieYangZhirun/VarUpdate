## 一、构建与工程化

### 1.1 打包工具：esbuild

仓库内 TypeScript 包默认使用 **esbuild** 打包，输出 **ESM**。

#### esbuild 配置要点

```typescript
// VarUpdate 构建入口：esbuild.config.mjs
import { build } from 'esbuild';

build({
  entryPoints: ['src/index.ts'],
  bundle: true,
  outfile: 'dist/bundle.js',
  format: 'esm',               // ES Module 格式，支持顶层 import/export
  platform: 'browser',
  target: 'es2020',
  sourcemap: isDev,
  minify: !isDev,
  legalComments: 'external',   // 许可注释单独文件，减小 bundle
});
```

> `schema-to-zod`、`flexible-json-patch`、`promptal-yaml` 位于 `src/` 模块目录，作为普通 TypeScript 源码直接参与打包；`klona`、`smol-toml` 等 npm 依赖同样打入 bundle，不列 external。

> 输出 **ESM**：酒馆助手脚本 iframe 支持顶层 `import`/`export`，与 esbuild `format: 'esm'` 一致。

### 1.2 TypeScript 配置

```jsonc
// tsconfig.json
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "ESNext",
    "lib": ["ES2020", "DOM"],
    "moduleResolution": "bundler",       // esbuild 的模块解析模式
    "esModuleInterop": true,
    "allowSyntheticDefaultImports": true,
    "strict": true,                      // 开启严格模式
    "skipLibCheck": true,
    "resolveJsonModule": true,           // 支持 import JSON 文件
    "declaration": false,                // 不需要 .d.ts 输出
    "sourceMap": true,
    "baseUrl": ".",
    "paths": {
      "@/*": ["./src/*"]                 // 路径别名
    }
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules"]
}
```

### 1.3 依赖引入策略

#### 四种引入方式

| 来源 | 引入方式 | 包含 |
|------|---------|------|
| **iframe 全局** | 直接使用，不需任何安装 | `_`(lodash)、`z`(Zod)、`YAML`(js-yaml)、`toastr`、`$`(jQuery)、`showdown` |
| **酒馆助手 API** | 全局函数/对象 | `TavernHelper`、`eventEmit/On/Once/ClearAll`、`SillyTavern` |
| **酒馆本体模块** | `TavernHelper.import()` 动态加载 | `extension_settings`、`promptManager`、世界书 API |
| **npm 依赖（打入 bundle）** | `npm install` + esbuild 打包 | `klona`、`smol-toml`、`json5` |

#### VarUpdate：所有源码与 npm 依赖一并打包

`schema-to-zod`、`flexible-json-patch`、`promptal-yaml` 位于 `src/` 模块目录，作为项目自身源码直接打包；其余 npm 依赖同样打入 bundle。

> **Zod 使用全局 `z`**：Zod 由酒馆助手预注入到 iframe 全局（`window.z`），VarUpdate 直接使用全局 `z`，**不打包 Zod 到 bundle**。`zod` 仅作为 `devDependencies`，为 TypeScript 提供类型推导；运行时类型声明在 `src/types/index.ts` 的 `declare global` 块中定义。

```typescript
// VarUpdate 中的实际引用方式（相对路径，无 alias）：
import { compileSchema } from './schema-to-zod.js';           // schema-compiler 内部
import { parseInstructions } from './flexible-json-patch.js'; // json-patch 内部
import { serializeToPromptalYAML } from '../shared/promptal-yaml.js'; // macro-engine

// Zod 使用 iframe 全局 z，无需 import：
// declare const z: typeof import('zod');  ← 见 src/types/index.ts declare global

// 其他 npm 依赖直接打入 bundle：
import { klona } from 'klona/full';
import { parse as parseToml } from 'smol-toml';
```

#### iframe 全局变量声明

> 全局变量声明集中在 `src/types/index.ts` 的 `declare global` 块中，为 TypeScript 编译器提供类型支持。运行时这些全局变量由酒馆助手的 `predefine.js` 注入。

```typescript
// src/types/index.ts — declare global 块（摘录）
declare global {
  /** lodash（可选，精简类型） */
  const _: { cloneDeep?: <T>(obj: T) => T; [key: string]: unknown } | undefined;

  const YAML: { load(input: string, options?: unknown): unknown; dump?(input: unknown, options?: unknown): string };

  const toastr: {
    success(message: string, title?: string, options?: unknown): void;
    error(message: string, title?: string, options?: unknown): void;
    warning(message: string, title?: string, options?: unknown): void;
    info(message: string, title?: string, options?: unknown): void;
  };

  const $: unknown;

  /** Zod（由酒馆助手注入） */
  const z: typeof import('zod');

  function eventEmit(eventName: string, ...args: unknown[]): Promise<void>;
  function eventOn(eventName: string, handler: (...args: unknown[]) => void): { stop: () => void };
  function eventOnce(eventName: string, handler: (...args: unknown[]) => void): { stop: () => void };
  function eventClearAll(): void;
  function eventRemoveListener(eventName: string, handler: (...args: unknown[]) => void): void;

  // 变量操作、宏注册、按钮等 API 详见实际代码
}
```

> **注意**：klona 不在全局列表中，VarUpdate 通过 npm 依赖 `klona/full` 打入 bundle 使用。

### 1.4 开发工作流

```bash
# 安装开发依赖（esbuild、typescript 等构建工具）
npm install

# 开发构建（含 sourcemap，不压缩）
npm run dev
# 各子项目若配置了 dev:watch 可监听重建；VarUpdate 当前仅有 dev / build，无 dev:watch。

# 测试流程
# 1. 构建生成 dist/bundle.js
# 2. 将 bundle.js 放入酒馆助手的脚本目录
# 3. 在酒馆中刷新页面，脚本自动加载
# 4. 打开浏览器 DevTools 查看控制台输出

# 发布：npm run build 后将 dist/bundle.js 部署到酒馆助手的脚本目录
```

## 二、错误处理模式

### 2.1 错误分级

所有错误按**后果严重性**分为三级：

| 等级 | 名称 | 后果 | 处理方式 | 示例 |
|------|------|------|---------|------|
| **L1** | 静默容错 | 不影响核心流程 | `notify('debug', ...)` + 继续执行 | 单条 Patch 指令校验失败（丢弃该条，继续后续） |
| **L2** | 通知用户 | 影响当前步骤但不致命 | `notify()` + 继续/重试 | API 调用超时（触发重试）、Schema 编译警告 |
| **L3** | 中止流程 | 无法继续的致命错误 | `feedback()` + 抛出异常中止 | 所有重试耗尽、配置数据损坏、API Key 无效 |

### 2.2 自定义错误类

两个项目共享相同的错误类层次结构：

```typescript
/**
 * 所有自定义错误的基类
 * 附带 context 字段用于结构化错误信息
 */
class ScriptError extends Error {
  constructor(
    message: string,
    public readonly context: Record<string, any> = {}
  ) {
    super(message);
    this.name = this.constructor.name;
  }
}

// ─── Agents 专用 ───
class ApiCallError extends ScriptError {}      // API 调用失败（网络/认证/格式）
class PipelineAbortError extends ScriptError {} // 流水线被中止（用户手动/重试耗尽）
class ConfigError extends ScriptError {}        // 配置数据无效

// ─── VarUpdate 专用 ───
class FormatParseError extends ScriptError {}   // YAML/JSON/TOML 均无法解析
class SchemaCompileError extends ScriptError {} // Schema 编译失败
class PatchParseError extends ScriptError {}    // Patch 指令解析失败（执行阶段通过 discard 机制处理，不抛自定义异常）
```

### 2.3 catch 策略

#### 规则 1：在边界处 catch，不在内部 catch

模块内部的函数**不应自行 catch 异常**，而是让异常向上冒泡到**调用方**。调用方根据上下文决定如何处理。

```typescript
// ❌ 错误做法：模块内部吞掉异常
function parseText(text: string) {
  try {
    return JSON.parse(text);
  } catch {
    return null;  // 吞掉了错误信息
  }
}

// ✅ 正确做法：模块内部抛出，调用方处理
function parseText(text: string): Record<string, any> {
  // 尝试 JSON → TOML → YAML，全部失败则抛出
  throw new FormatParseError('三种格式均无法解析', { jsonErr, tomlErr, yamlErr });
}

// 调用方（主控制器）
try {
  const data = parseText(rawText);
} catch (e) {
  if (e instanceof FormatParseError) {
    notify('error', '格式解析失败', e.message);
    return;  // 不中止整个流程，跳过这条消息
  }
  throw e;  // 未知异常继续向上冒泡
}
```

#### 规则 2：只 catch 你知道如何处理的异常

```typescript
try {
  await apiCall(config);
} catch (e) {
  if (e instanceof ApiCallError) {
    // 已知异常：按重试逻辑处理
    retryController.handleFailure(e);
  } else {
    // 未知异常：不吞，向上抛出
    throw e;
  }
}
```

#### 规则 3：async 函数中 catch 后不要忘记 return 或 throw

```typescript
// ❌ catch 后没有 return，函数继续执行后续逻辑
async function process() {
  try {
    await step1();
  } catch (e) {
    notify('error', '步骤1失败', e.message);
    // 缺少 return! step2 会继续执行
  }
  await step2();
}

// ✅ catch 后明确 return
async function process() {
  try {
    await step1();
  } catch (e) {
    notify('error', '步骤1失败', e.message);
    return;  // 明确中止
  }
  await step2();
}
```

### 2.4 错误传播路线

```text
模块内部（抛出 ScriptError 子类）
  ↓ 冒泡
模块公开函数（不 catch，让异常穿透）
  ↓ 冒泡
主控制器 / 流程编排层（catch 并决策）
  ├─ L1：warn + 继续
  ├─ L2：notify + 重试 / 跳过
  └─ L3：feedback + 中止流程
```

## 三、模块间通信模式

### 3.1 同脚本内部：直接 import

同一脚本内的模块之间使用 **ES Module import** 直接调用，不走事件总线。

```typescript
// ✅ 脚本内部模块间：直接 import
import { parseStructuredText } from './modules/format-parser';
import { notify } from './modules/notification';

const data = parseStructuredText(rawText);
notify('debug', '解析完成', `得到 ${Object.keys(data).length} 个字段`);
```

事件总线 **仅用于跨脚本通信**（Agents ↔ VarUpdate）和**酒馆本体事件监听**。脚本内部不使用事件解耦（过度解耦会增加调试难度、破坏类型安全）。

### 3.2 异步处理：统一 async/await

所有异步操作使用 **async/await**，不使用 `.then()` 链。

```typescript
// ✅ async/await
async function runStep(agent: AgentConfig): Promise<StepResult> {
  const prompt = await buildPrompt(agent);
  const response = await callApi(agent, prompt);
  const processed = postProcess(response, agent.postProcess);
  return processed;
}

// ❌ .then() 链
function runStep(agent: AgentConfig) {
  return buildPrompt(agent)
    .then(prompt => callApi(agent, prompt))
    .then(response => postProcess(response, agent.postProcess));
}
```

**例外**：事件监听回调中的异步操作可使用 void async IIFE：

```typescript
eventBus.on('agents:message_complete', (payload) => {
  void (async () => {
    await processMessage(payload);
  })();
});
```

### 3.3 状态管理：模块级单例

每个有状态的模块使用**模块级变量**维护状态（利用 ES Module 的单例特性），对外暴露 getter/setter 函数：

```typescript
// modules/notification.ts
let currentLevel: NotifyLevel = 'notice';

export function setLevel(level: NotifyLevel): void {
  currentLevel = level;
}

export function notify(level: NotifyLevel, title: string, msg: string): void {
  if (levelValue(level) < levelValue(currentLevel)) return;
  // ...
}
```

不使用 class 单例模式（class 增加了不必要的 `this` 绑定复杂度），不使用全局变量（避免跨模块耦合）。

### 3.4 初始化顺序

主控制器 `index.ts` 负责按依赖顺序初始化所有模块：

```typescript
// src/index.ts
export async function init() {
  // 1. 无依赖模块先初始化
  notificationModule.setLevel(globalSettings.notifyLevel);

  // 2. 注册事件监听
  eventBus.setupListeners();

  // 3. 加载持久化配置
  await configStore.load();

  // 4. UI 最后初始化（依赖配置数据）
  uiPanel.render();
}
```

## 四、命名约定

### 4.1 文件名

| 类型 | 规则 | 示例 |
|------|------|------|
| 模块文件 | `kebab-case.ts` | `label-engine.ts`、`format-parser.ts` |
| 类型定义 | `kebab-case.ts`（置于 `types/`） | `types/index.ts` |
| 数据文件 | `kebab-case.json` | `api-params-map.json` |
| 目录 | `kebab-case` | `schema-compiler/`、`json-patch/` |

### 4.2 代码命名

| 元素 | 规则 | 示例 |
|------|------|------|
| 变量 / 函数 | `camelCase` | `parseText`、`currentLevel` |
| 类 / 接口 / 类型 | `PascalCase` | `AgentConfig`、`SchemaCompileError` |
| 常量 | `UPPER_SNAKE_CASE` | `SCHEMA_KEYWORDS`、`OP_ALIASES` |
| 事件名 | `namespace:snake_case` | `agents:pipeline_started`、`varupdate:updated` |
| 存储键名 | `PascalCase_snake_case` 混合 | `Agents_config`、`VarUpdate_config` |

### 4.3 注释

- **公共函数/接口**：必须有 JSDoc 注释，包含 `@param` 和 `@returns`
- **模块顶部**：简述该模块的职责、上下游依赖
- **复杂逻辑**：行内注释解释「为什么」而非「做了什么」
- **TODO / FIXME**：格式 `// TODO(模块名): 描述`

### 4.4 导出规则

- 每个模块导出**具名函数**（`export function`），不导出 default
- 模块目录（如 `schema-compiler/`）通过 `index.ts` 统一导出
- 类型定义通过 `export type` / `export interface` 导出

```typescript
// ✅ modules/schema-compiler/index.ts
export { compileSchema } from './compiler';
export { validateWithSchema } from './validator';
export type { CompiledSchema, ValidationResult } from './types';
```
